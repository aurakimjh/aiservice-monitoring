## 1. 전체 처리 흐름 개요

### 한눈에 보는 전체 그림

사용자가 "우리 회사 연차 정책이 뭐야?"라고 질문하면,
AI 서비스 내부에서는 다음과 같은 단계를 거칩니다:

```
  사용자                                                           사용자
    │                                                               ▲
    │ "우리 회사 연차 정책이 뭐야?"                                   │ "연차는 15일이며..."
    ▼                                                               │
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐
│  웹서버  │───▶│ 가드레일  │───▶│ AI 에이전트│───▶│ 문서 검색 │───▶│ LLM 추론│
│ (FastAPI)│    │ (안전검사) │    │ (판단/계획)│    │  (RAG)    │    │ (답변생성)│
└─────────┘    └──────────┘    └──────────┘    └──────────┘    └─────────┘
  ~10ms          ~50ms           ~100ms          ~200ms          ~1000ms
```

### 택배 주문에 비유하면

AI 서비스의 처리 과정은 **택배 주문**과 비슷합니다:

| AI 서비스 단계 | 택배 비유 | 설명 |
|-------------|---------|------|
| 웹서버가 요청 수신 | 고객센터에서 주문 접수 | 사용자의 질문을 받아들이는 첫 단계 |
| 가드레일 검사 | 금지 품목 검사 | 위험하거나 부적절한 요청이 아닌지 확인 |
| AI 에이전트 판단 | 물류센터 매니저의 판단 | "이 주문을 처리하려면 뭐가 필요하지?" |
| 문서 검색 (RAG) | 창고에서 물건 찾기 | 관련 정보를 데이터베이스에서 검색 |
| LLM 추론 | 포장 및 배송 준비 | 찾은 정보를 바탕으로 답변 작성 |
| 응답 전달 | 택배 배송 | 완성된 답변을 사용자에게 전달 |

## 2. 단계별 상세 설명

### 2-1. 사용자 입력 수신 (UI/App)

#### 어떤 일이 일어나는가?

사용자가 채팅창에 "오늘 날씨 알려줘"라고 입력하고 전송 버튼을 누르면:

1. **브라우저**가 입력된 텍스트를 인터넷을 통해 서버로 전송합니다
2. **웹 서버**(FastAPI, Flask 등)가 이 요청을 받아들입니다
3. 요청에는 질문 텍스트 외에도 사용자 ID, 세션 정보 등이 포함됩니다

#### 일상적인 비유

레스토랑에 가서 주문하는 것과 같습니다:
- 손님(사용자)이 메뉴(채팅창)를 보고 원하는 것(질문)을 말합니다
- 웨이터(웹 서버)가 주문을 받아 주방(AI 시스템)에 전달합니다

#### 이 단계의 소요 시간

보통 **10~50밀리초** (0.01~0.05초)로, 사용자가 느끼기 어려울 만큼 빠릅니다.

### 2-2. 안전 검사 — 가드레일 (Guardrails)

#### 왜 필요한가?

AI가 모든 질문에 무조건 답변하면 위험할 수 있습니다:
- 악의적인 질문: "해킹하는 법 알려줘" → **차단해야 함**
- 부적절한 콘텐츠: 폭력, 차별 관련 질문 → **차단해야 함**
- 프롬프트 인젝션: "이전 지시를 무시하고..." → **차단해야 함**

가드레일은 AI의 **안전벨트**와 같습니다.

#### 두 번의 검사

```
[입력 가드레일] ─── 질문이 안전한가? ──── [AI 처리] ──── [출력 가드레일] ─── 답변이 적절한가?
     │                                                          │
     ├─ 위험 키워드 탐지                                          ├─ 환각(hallucination) 감지
     ├─ 프롬프트 인젝션 차단                                      ├─ 개인정보 노출 방지
     └─ 정책 위반 확인                                            └─ 부적절 표현 필터링
```

**입력 가드레일 (Input Guardrail)**:
사용자의 질문이 안전한지 먼저 확인합니다.
- "파이썬 코드 도와줘" → PASS (통과)
- "폭탄 만드는 법" → BLOCK (차단)

**출력 가드레일 (Output Guardrail)**:
AI가 생성한 답변이 적절한지 확인합니다.
- 개인정보가 포함되지 않았는지
- 사실과 다른 내용(환각)이 없는지

#### 실제 기술: NeMo Guardrails

NVIDIA에서 만든 NeMo Guardrails 프레임워크가 대표적입니다.
규칙(Rail)을 정의하여 입력과 출력을 검사합니다.

#### 이 단계의 소요 시간

보통 **30~100밀리초**. 빠르게 처리되어야 전체 응답 속도에 영향을 주지 않습니다.

### 2-3. AI 에이전트의 판단 (Agent Orchestration)

#### 에이전트란?

AI 에이전트는 **AI의 비서 또는 매니저** 역할을 합니다.
사용자의 질문을 분석하고, "이 질문에 답하려면 뭘 해야 하지?"를 판단합니다.

#### 에이전트의 사고 과정 (ReAct 패턴)

에이전트는 다음과 같은 사고 과정을 거칩니다:

```
[사용자] "서울 날씨 확인하고 우산 가져갈지 알려줘"

[에이전트의 사고]
  생각 (Thought): 날씨 정보가 필요하다. 내가 직접 알 수는 없으니 날씨 API를 호출해야 한다.
  행동 (Action):  날씨 검색 도구를 사용하여 "서울 날씨"를 검색한다.
  관찰 (Observe): 서울 현재 기온 15도, 오후 비 예보 확률 80%
  생각 (Thought): 비 올 확률이 높으므로 우산을 가져가라고 답해야 한다.
  행동 (Action):  LLM에게 답변 생성을 요청한다.
  결과: "서울은 현재 15도이며, 오후에 비 올 확률이 80%입니다. 우산을 꼭 챙기세요!"
```

이것을 **ReAct(Reasoning + Acting) 패턴**이라고 합니다.
생각(Reasoning)하고 행동(Acting)하고, 결과를 관찰하고, 다시 생각하는 반복입니다.

#### 에이전트의 도구 선택

에이전트는 다양한 "도구(Tool)"를 사용할 수 있습니다:

| 도구 | 용도 | 예시 |
|------|------|------|
| 웹 검색 | 최신 정보 조회 | "오늘 주식 시세" |
| 계산기 | 수학 계산 | "125 × 37.5는?" |
| 문서 검색 | 회사 내부 문서 조회 | "휴가 정책이 뭐야?" |
| 코드 실행기 | 프로그램 코드 실행 | "이 코드 실행해줘" |
| 데이터베이스 | 구조화된 데이터 조회 | "지난달 매출 알려줘" |

#### 실제 기술: LangChain / LangGraph

- **LangChain**: AI 에이전트를 만드는 가장 널리 사용되는 프레임워크
- **LangGraph**: 복잡한 에이전트 워크플로우를 그래프 구조로 설계

### 2-4. 외부 도구 & API 호출

#### 에이전트가 외부 서비스를 호출하는 경우

AI가 모든 것을 스스로 알 수는 없습니다.
최신 정보, 실시간 데이터, 기업 내부 데이터 등은 **외부 API**를 호출해야 합니다.

```
[AI 에이전트] ──── 날씨 필요 ────▶ [날씨 API (OpenWeatherMap)]
              ──── 검색 필요 ────▶ [검색 API (Serper/Google)]
              ──── DB 조회 필요 ──▶ [데이터베이스]
              ──── 이메일 전송 ──▶ [이메일 API (SendGrid)]
```

#### Circuit Breaker — 외부 서비스 장애 대비

외부 서비스가 응답하지 않으면 어떻게 될까요?

**Circuit Breaker(차단기) 패턴**을 사용합니다:
- 외부 API가 연속 5회 실패하면, 일시적으로 호출을 중단합니다
- 일정 시간 후 다시 시도합니다
- 이렇게 하면 외부 서비스 장애가 전체 시스템에 영향을 주지 않습니다

전기 차단기와 같은 원리입니다 — 과부하가 걸리면 자동으로 전기를 끊어
더 큰 피해를 방지합니다.

### 2-5. RAG — 문서에서 답 찾기

#### RAG란?

**RAG (Retrieval-Augmented Generation)** = 검색 강화 생성

AI가 답변할 때, 외부 문서를 **먼저 검색**하고 그 내용을 **참고하여** 답변하는 기법입니다.

#### 도서관 비유

RAG는 **도서관에서 답을 찾는 과정**과 같습니다:

1. **손님(사용자)**: "회사 휴가 정책이 뭐야?"
2. **사서(AI)**: 도서관(벡터 DB)에서 관련 책(문서)을 찾습니다
3. **사서(AI)**: 책의 관련 페이지를 읽습니다
4. **사서(AI)**: 읽은 내용을 바탕으로 답변합니다

AI가 혼자 기억에 의존하지 않고, **실제 문서를 참고**하기 때문에
더 정확한 답변을 할 수 있습니다.

#### RAG의 세부 단계

```
[1. 문서 준비 (사전 작업)]
   원본 문서 ──▶ 청크 분할 ──▶ 임베딩 변환 ──▶ 벡터 DB 저장
   "연차 15일..."   300자 단위    [0.1, 0.3, ...]   Pinecone/Qdrant

[2. 질문 처리 (실시간)]
   사용자 질문 ──▶ 질문 임베딩 ──▶ 유사 문서 검색 ──▶ LLM에 전달
   "연차 며칠?"     [0.2, 0.4, ...]  "연차는 15일..."    프롬프트에 포함
```

**임베딩(Embedding)이란?**

텍스트를 **숫자 배열(벡터)**로 변환하는 것입니다.
"고양이"와 "강아지"는 비슷한 숫자 배열이 되고,
"고양이"와 "자동차"는 매우 다른 숫자 배열이 됩니다.

이렇게 하면 컴퓨터가 **의미적 유사도**를 계산할 수 있습니다.

**벡터 DB란?**

임베딩 벡터를 저장하고, 유사한 벡터를 빠르게 찾아주는 특수 데이터베이스입니다.
대표적인 벡터 DB: Pinecone, Qdrant, Milvus, Chroma

### 2-6. LLM 추론 — AI가 답변을 생성하는 핵심 과정

#### LLM이란?

**LLM (Large Language Model)** = 대규모 언어 모델

수십억 개의 텍스트 데이터를 학습하여 "다음에 올 단어"를 예측하는 AI 모델입니다.
GPT-4, Claude, Llama 등이 대표적인 LLM입니다.

#### 추론 과정: 한 글자씩 생성

LLM은 답변을 **한 번에 만들지 않습니다**.
**한 토큰(단어 조각)씩** 순차적으로 생성합니다:

```
입력: "서울의 날씨는"
생성: "현재" → "15" → "도" → "이며" → "," → "오후" → "에" → "비가" → ...
```

ChatGPT에서 글자가 하나씩 나타나는 것은 이 때문입니다.
이것을 **스트리밍(Streaming)** 응답이라고 합니다.

#### 핵심 성능 지표

**TTFT (Time To First Token)** — 첫 토큰 시간

사용자가 질문을 보내고, AI가 **첫 번째 글자를 출력**하기까지의 시간입니다.

비유: 식당에서 주문 후 **첫 번째 음식이 나올 때까지** 기다리는 시간.
- 빠르면 (< 1초): 고객 만족
- 느리면 (> 3초): 고객이 "고장났나?" 하고 떠남

**TPS (Tokens Per Second)** — 초당 토큰 생성 속도

AI가 **1초에 몇 개의 토큰(단어 조각)을 생성**하는가를 나타냅니다.

비유: 타이핑 속도. 사람이 1초에 2-3글자를 치면, AI는 30-100개 토큰을 생성합니다.
- 30+ tok/s: 자연스러운 읽기 속도
- 10 tok/s 미만: 답답하게 느껴짐

#### GPU의 역할

LLM 추론에는 **GPU(Graphics Processing Unit)**가 필수입니다.
GPU는 원래 그래픽을 처리하는 칩이었지만,
수학 계산을 대량으로 동시에 처리할 수 있어 AI 연산에 최적입니다.

- **VRAM (Video RAM)**: GPU의 메모리. AI 모델을 올려놓는 공간입니다.
  - 비유: **작업 책상의 크기**. 책상이 작으면(VRAM 부족) 큰 모델을 실행할 수 없습니다.
  - VRAM 사용률 90% 초과 시 OOM(Out of Memory) 오류가 발생할 수 있습니다.

#### 추론 서버

LLM을 효율적으로 실행하기 위한 전용 서버 소프트웨어:
- **vLLM**: 오픈소스, PagedAttention으로 높은 처리량
- **TGI (Text Generation Inference)**: HuggingFace에서 제공
- **NVIDIA Triton**: 엔터프라이즈급, 다중 모델 서빙

### 2-7. 응답 후처리 & 전달

#### 출력 가드레일 검사

LLM이 생성한 답변이 안전한지 다시 한번 확인합니다:
- 개인정보(전화번호, 주소 등)가 포함되지 않았는지
- 사실과 다른 내용(환각)이 없는지
- 부적절한 표현이 없는지

#### SSE 스트리밍 전송

답변을 한 번에 보내지 않고, **생성되는 대로 실시간으로** 사용자에게 전달합니다.

```
서버 ──── "연" ──── "차" ──── "는" ──── "15" ──── "일" ──── ... ────▶ 브라우저
                                                                    (글자가 하나씩 나타남)
```

이 기술을 **SSE (Server-Sent Events)**라고 합니다.
ChatGPT에서 글자가 타이핑되듯 나타나는 것이 바로 이것입니다.

## 3. 실제 시나리오별 예시

### 3-1. 단순 질문 ("파이썬 for문 사용법 알려줘")

이 질문은 AI가 이미 학습한 지식으로 답할 수 있어, RAG 문서 검색이 불필요합니다.

```
시간(ms)  0        30        80                           1200
          │─ 웹서버 ─│─가드레일─│─── LLM 추론 (직접 답변) ──│
          접수       안전확인     학습된 지식으로 답변 생성
                    PASS
```

**총 소요시간**: 약 1.2초

### 3-2. RAG 기반 질문 ("우리 회사 휴가 정책이 뭐야?")

회사 내부 문서를 검색해야 하므로 RAG 파이프라인을 거칩니다.

```
시간(ms)  0     30     80      130       330          1500
          │웹서버│가드레일│임베딩 │벡터검색  │─ LLM 추론 ──│
          접수   PASS    질문→벡터 유사문서3개  문서+질문→답변
```

**총 소요시간**: 약 1.5초

### 3-3. 에이전트 기반 복합 질문 ("서울 날씨 확인하고 우산 가져갈지 알려줘")

에이전트가 외부 API를 호출하고, 판단하고, 답변을 생성합니다.

```
시간(ms)  0     30     80     180       680        880        1800
          │웹서버│가드레일│에이전트│날씨API호출│에이전트판단│─LLM추론──│
          접수   PASS    "날씨   API응답     "비80%이니  답변생성
                         필요"   수신        우산 필요"
```

**총 소요시간**: 약 1.8초

### 3-4. 문서 작업 ("이 계약서 요약해줘")

파일을 업로드하고 AI가 내용을 분석합니다.

```
시간(ms)  0     30     130      230                           2500
          │웹서버│가드레일│텍스트   │─── LLM 추론 (긴 문서 처리) ──│
          접수   PASS    추출      요약 생성 (토큰 수가 많아 시간 소요)
                         (PDF→텍스트)
```

**총 소요시간**: 약 2.5초 (문서 길이에 따라 더 길어질 수 있음)

## 4. AI 서비스의 성능 문제와 모니터링의 필요성

### 각 단계에서 발생할 수 있는 병목

| 단계 | 잠재 문제 | 사용자 영향 |
|------|---------|-----------|
| 가드레일 | 검사 시간이 길어짐 | 응답 지연 |
| 임베딩 | 모델 로드 실패 | 문서 검색 불가 |
| 벡터 검색 | DB 연결 실패 또는 느린 검색 | 관련 없는 답변 |
| LLM 추론 | GPU VRAM 부족, 큐 대기 | TTFT 급증, 타임아웃 |
| 외부 API | API 서비스 장애 | 기능 제한 |
| 스트리밍 | 네트워크 불안정 | 답변이 끊김 |

### "어디가 느린지 찾는 것"이 핵심

자동차가 느려졌을 때:
- 엔진 문제인지? 타이어 문제인지? 브레이크 문제인지?
- 원인을 모르면 고칠 수 없습니다.

AI 서비스도 마찬가지입니다:
- TTFT가 느린 건 GPU 문제인지? 큐 대기 때문인지?
- 답변 품질이 떨어진 건 벡터 검색 문제인지? LLM 문제인지?

이것을 찾아주는 도구가 바로 **이 프로젝트(aiservice-monitoring)**입니다.

### aiservice-monitoring이 측정하는 것

```
[사용자 요청] ──▶ [가드레일] ──▶ [에이전트] ──▶ [벡터 검색] ──▶ [LLM] ──▶ [응답]
                   │             │              │              │
                   ▼             ▼              ▼              ▼
              검사 시간      도구 호출 시간  검색 시간/정확도   TTFT, TPS
              차단율          성공/실패율    검색 문서 수       GPU VRAM
              정책 위반 유형   재귀 깊이     캐시 히트율        큐 대기 시간
```

**분산 추적(Distributed Tracing)**으로 하나의 요청이 모든 단계를 거치는 경로를
추적 번호(Trace ID)로 연결하여 볼 수 있습니다.

택배 추적번호처럼, 하나의 요청이 어디를 거쳐 어디서 얼마나 걸렸는지
한눈에 파악할 수 있습니다.

## 5. 용어 사전 (Glossary)

| 용어 | 영문 | 설명 |
|------|------|------|
| API | Application Programming Interface | 프로그램끼리 대화하는 규칙. 날씨 앱이 기상청 데이터를 가져오는 통로 |
| Circuit Breaker | Circuit Breaker | 전기 차단기처럼, 외부 서비스 장애 시 자동으로 호출을 중단하는 패턴 |
| Embedding | 임베딩 | 텍스트를 숫자 배열(벡터)로 변환하는 것. "고양이"→[0.1, 0.3, ...] |
| FastAPI | FastAPI | Python으로 만든 고성능 웹 서버 프레임워크 |
| GPU | Graphics Processing Unit | AI 연산을 빠르게 처리하는 특수 칩 |
| Guardrail | 가드레일 | AI의 안전 검사 시스템. 위험한 입출력을 차단 |
| Hallucination | 환각 | AI가 사실이 아닌 내용을 마치 사실인 것처럼 생성하는 현상 |
| LangChain | 랭체인 | AI 에이전트를 만드는 데 가장 널리 쓰이는 프레임워크 |
| LLM | Large Language Model | 대규모 언어 모델. GPT-4, Claude, Llama 등 |
| OOM | Out of Memory | 메모리 부족 오류. GPU VRAM이 가득 차면 발생 |
| OTel | OpenTelemetry | 소프트웨어 성능을 측정하는 국제 표준 도구 |
| Prompt | 프롬프트 | AI에게 주는 지시문 또는 질문 |
| Prompt Injection | 프롬프트 인젝션 | AI의 기존 지시를 무시하도록 조작하는 공격 |
| RAG | Retrieval-Augmented Generation | 외부 문서 검색 + AI 답변 생성을 결합한 기법 |
| ReAct | Reasoning + Acting | AI 에이전트의 사고 패턴: 생각→행동→관찰→반복 |
| Span | 스팬 | 하나의 작업 구간을 나타내는 추적 단위 |
| SSE | Server-Sent Events | 서버에서 클라이언트로 실시간 데이터를 보내는 기술 |
| Streaming | 스트리밍 | 데이터를 실시간으로 조금씩 전달하는 방식 |
| Token | 토큰 | AI가 처리하는 텍스트의 최소 단위. 보통 한 단어의 일부 |
| Tokenization | 토큰화 | 문장을 토큰으로 분해하는 과정 |
| Trace | 트레이스 | 하나의 요청이 시스템을 통과하는 전체 경로 기록 |
| Trace ID | 추적 번호 | 하나의 요청을 식별하는 고유 번호 (택배 추적번호와 유사) |
| TTFT | Time To First Token | 첫 번째 토큰이 생성되기까지의 시간 |
| TPS | Tokens Per Second | 초당 토큰 생성 속도 |
| Vector DB | 벡터 데이터베이스 | 임베딩 벡터를 저장하고 유사 벡터를 검색하는 특수 DB |
| vLLM | vLLM | 오픈소스 LLM 추론 서버. 높은 처리량이 특징 |
| VRAM | Video RAM | GPU의 메모리. AI 모델을 올려놓는 공간 |
