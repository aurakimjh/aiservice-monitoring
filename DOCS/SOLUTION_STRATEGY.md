### 1.2 미션 — 우리가 매일 하는 일

**"AI 서비스를 운영하는 모든 팀이 장애를 예방하고, 성능을 최적화하며, 비용을 절감할 수 있도록 한다."**

AITOP은 세 가지 약속을 지킨다:

| 약속 | 의미 | 측정 지표 |
|------|------|----------|
| **1. 즉각적 인지** | 문제가 발생하면 30초 이내에 알린다 | MTTD (Mean Time To Detect) < 30s |
| **2. 정확한 진단** | 근본 원인을 5분 이내에 특정한다 | MTTR (Mean Time To Resolve) < 5min |
| **3. 선제적 예방** | 장애가 발생하기 전에 예측하여 대응한다 | Predictive Accuracy > 85% |

## 2. 시장 분석 (2026)

### 2.1 AI Observability 시장 개요

2026년은 AI Observability가 독립된 시장 카테고리로 확립되는 원년이다. 기존 APM 벤더들이 "AI 모니터링 기능 추가"로 대응하고 있지만, AI-Native 접근 방식과는 근본적인 차이가 존재한다.

```
AI Observability 시장 진화 타임라인
═══════════════════════════════════════════════════════════════════════

2020 ─── APM 전성기 (Datadog IPO $38B)
  │       전통적 APM: 웹 서비스 + 마이크로서비스 중심
  │
2022 ─── ChatGPT 등장, LLM 시대 개막
  │       AI 모니터링 수요 폭발적 증가
  │
2023 ─── AI Observability 태동
  │       Arize AI Series B $38M, LangSmith 런칭
  │       APM 벤더들 "AI Monitoring" 베타 출시
  │
2024 ─── 시장 분화 시작
  │       AI-Native vs Traditional APM+AI 진영 분리
  │       OTel Semantic Convention for GenAI 초안 발표
  │
2025 ─── 시장 가속
  │       Enterprise AI 도입 본격화, GPU 인프라 투자 급증
  │       AI 서비스 장애 사례 증가 → 모니터링 필수화
  │
2026 ─── 시장 확립 ← 우리는 여기에 있다
  │       AI Observability TAM $12B 돌파
  │       규제 요구 (EU AI Act) → 관측성 의무화 흐름
  │       ★ AITOP 시장 진입 최적 시점
  │
2028 ─── 시장 성숙
          AI Observability = 기본 인프라
          자동화(AIOps) + 예측(Predictive) 경쟁
```

### 2.2 시장 규모 (TAM / SAM / SOM)

| 지표 | 2025년 | 2026년 | 2028년 | CAGR |
|------|--------|--------|--------|------|
| **TAM** (Total Addressable Market) | $8.2B | $12.1B | $24.6B | 38% |
| **SAM** (Serviceable Available Market) | $2.4B | $3.8B | $8.1B | 40% |
| **SOM** (Serviceable Obtainable Market) | — | $15M | $120M | — |

**TAM 산출 근거**:
- 글로벌 Observability 시장: $58B (2026, Gartner)
- AI/ML 워크로드 비중: 21% (2026년 예측)
- $58B x 21% = $12.18B

**SAM 산출 근거**:
- APAC + 자체 호스팅 선호 시장 집중
- On-Prem / Hybrid 배포 수요: 전체의 31%
- $12.1B x 31% = $3.8B

**SOM 산출 근거**:
- Year 1 한국 시장: Enterprise AI 기업 200개사 x $6,000 평균 연간 계약 = $1.2M
- Year 1 Lite 컨설팅: 500 프로젝트 x $3,000 = $1.5M
- Year 2~3 APAC 확장: $15M → $120M (일본, 동남아, 호주)

## 5. 기술 전략

### 5.1 아키텍처 철학 — 왜 이 설계가 최선인가

AITOP의 기술 아키텍처는 다섯 가지 원칙 위에 세워졌다:

```
AITOP 아키텍처 5대 원칙
═══════════════════════════════════════════════════════════════════

  원칙 1: OpenTelemetry First — 표준이 곧 경쟁력
  ─────────────────────────────────────────────────
  • 모든 데이터 수집/전송은 OTel Protocol (OTLP) 기반
  • OTel Semantic Convention for GenAI 선도적 구현
  • 고객의 기존 OTel 투자를 100% 활용 → 전환 비용 ZERO
  • 벤더 락인 없음 = 고객 신뢰 = 장기 관계

  원칙 2: Agent + Agentless Hybrid — 최적의 수집 전략
  ─────────────────────────────────────────────────
  • Agent (Go binary): OS/GPU/Process 레벨의 깊은 수집
  • Agentless (OTel SDK): 애플리케이션 레벨 계측
  • 고객이 선택: "Agent만", "OTel만", "둘 다" 모두 가능
  • 에어갭 환경도 Agent collect-export 모드로 지원

  원칙 3: AGPL-free Commercial Stack — 상용 라이선스 안전
  ─────────────────────────────────────────────────
  • Grafana (AGPL) → AITOP 자체 UI (Next.js 16, Apache 2.0)
  • Loki (AGPL) → 자체 로그 수집 파이프라인
  • 모든 의존성 Apache 2.0 / MIT / BSD 검증 완료
  • 고객의 법무팀이 안심할 수 있는 상용 스택

  원칙 4: Modular Collector Architecture — 확장 가능한 수집
  ─────────────────────────────────────────────────
  • 12종 Collector가 독립 모듈로 동작
  • 필요한 Collector만 활성화 → 리소스 최적화
  • 새로운 기술 등장 시 Collector 추가만으로 대응
  • 커뮤니티/파트너가 Collector 개발 가능 (Marketplace)

  원칙 5: Edge-to-Cloud Pipeline — 데이터 흐름 최적화
  ─────────────────────────────────────────────────
  • Agent에서 1차 집계 → 네트워크 트래픽 60% 절감
  • Tail-based Sampling → 의미 있는 트레이스만 저장
  • Hot/Warm/Cold 계층화 → 2년 보존에도 비용 최적화
  • 실시간 SSE 스트리밍 → 대시보드 지연 < 3초
```

```
Self-Healing 성숙도 모델

  Level 1: 자동 재시작 (기본)
  ─────────────────────────
  • 서비스 헬스체크 실패 → 자동 재시작

  • 느린 LLM 엔드포인트 → 트래픽 자동 Failover
  • 비정상 노드 → 자동 Drain + 교체

  Level 3: 자동 최적화 (고급)
  ─────────────────────────
  • GPU 메모리 단편화 → 자동 배치 재배치
  • 캐시 적중률 저하 → 캐시 정책 자동 조정
  • 모델 성능 저하 → 자동 모델 버전 롤백

  Level 4: 자율 운영 (미래)
  ─────────────────────────
  • 인시던트 발생 → 과거 이력 기반 자동 대응
  • 변경 사항 → 자동 영향 분석 + 안전 배포
  • 용량 예측 → 자동 인프라 프로비저닝
```

## 원클릭 프로파일링 전략 (Phase 33)

> **추가일**: 2026-03-25 | **연계**: [AGENT_DESIGN.md §14](./AGENT_DESIGN.md#14-runtime-attach-모듈--앱-재시작-없이-프로파일링)

### 핵심 차별화

경쟁사(Datadog APM, Dynatrace, Pinpoint)는 에이전트를 설치할 때 앱도 함께 재시작해야 한다.
AITOP은 **Runtime Attach** 방식으로 **앱 재시작 없이 즉시 프로파일링**을 활성화한다.

```
경쟁사 방식:
  에이전트 설치 → 앱 설정 변경 → [앱 재시작] → 프로파일링 시작
  ↑ 프로덕션 환경에서 재시작은 서비스 영향 (다운타임, 리스크)

AITOP 방식:
  에이전트 설치 (1회) → UI에서 "Java Profiler 배포" 클릭 → 프로파일링 시작 (재시작 없음)
```

### 원클릭 프로파일링 전체 워크플로

```
[1단계] 에이전트 설치 (1회)
  서버에 aitop-agent 바이너리 설치 → 이후 재설치 불필요

[2단계] UI에서 프로파일링 플러그인 배포
  /agents/{id} 화면 → "프로파일링 플러그인 배포" 버튼 클릭
  → 언어 선택 (Java / Python / .NET / Node.js / Go)
  → 서버 Plugin Registry에서 플러그인 파일 에이전트로 자동 전달

[3단계] 에이전트 자동 Attach
  에이전트가 대상 프로세스 PID 탐지 → Attach 방식으로 즉시 주입
  Java:   VirtualMachine.attach(pid) → loadAgent(aitop-java-agent.jar)
  Python: py-spy record --pid <pid>
  .NET:   DiagnosticsClient(pid) → EventPipe 세션 시작
  Node:   SIGUSR1 → CDP WebSocket 연결

[4단계] 프로파일링 데이터 수집 시작
  FlameGraph, Thread 덤프, GC 이벤트 → Collection Server → UI 실시간 표시
```

### 두 가지 프로파일링 모드

| | 🟢 Attach 모드 (즉시 활성화) | 🔴 Full Install 모드 (앱 재시작) |
|--|----------------------------|-------------------------------|
| **앱 재시작** | 불필요 | 필요 |
| **설정 반영** | 즉시 | 앱 재시작 시 |
| **프로덕션 적합성** | 긴급 진단, 단기 분석 | 상시 APM, 완전 계측 |
| **메소드 계측 범위** | 샘플링 + 런타임 로드 클래스 | 모든 클래스 (시작 시점 포함) |
| **Java** | JVM Attach API | `-javaagent` 옵션 |
| **Python** | py-spy 외부 샘플링 | sitecustomize 삽입 |
| **.NET** | EventPipe (GC/CPU/Thread) | CLR Profiler (전체 계측) |
| **Node.js** | CDP Inspector | `--require @aitop/profiler` |
