## 1. 설계 배경 및 목표

### 1.1 현재 문제점

| 문제 | 상세 | 영향 |
|------|------|------|
| UI 분산 | Grafana 5개 대시보드 + XLog 대시보드 + Jaeger + Prometheus가 별도 URL | 운영자가 4~5개 탭 전환 필수, 컨텍스트 유실 |
| 프로젝트 개념 부재 | 서비스별 대시보드가 있으나 프로젝트(고객/환경) 단위 관리 불가 | 멀티테넌트, 다중 고객 환경 대응 불가 |
| 자원 관리 부재 | 서버, 미들웨어, AI 시스템의 인벤토리 관리 UI 없음 | 어떤 서버에서 어떤 서비스가 돌고 있는지 한눈에 파악 불가 |
| 에이전트 연계 부재 | AITOP 에이전트에서 수집한 진단 데이터와 실시간 모니터링 연계 불가 | 진단 보고서와 모니터링 데이터 분리 |
| 상용 수준 미달 | 검색, 필터링, 커스텀 대시보드, 권한 관리 등 상용 기능 부재 | 솔루션 판매 불가 |

### 1.2 설계 목표

```
핵심 원칙: "한 화면에서 모든 것을 — One Pane of Glass"

1. 통합 중앙화    → 모든 모니터링 데이터를 단일 UI로 집중
2. 프로젝트 중심  → 고객/환경별 프로젝트로 리소스를 조직화
3. 계층적 드릴다운 → 프로젝트 → 호스트 그룹 → 서버 → 서비스 → 트레이스 → 스팬
4. 에이전트 통합  → AITOP Agent 수집 데이터 + OTel 실시간 데이터 통합 표시
5. 상용 품질     → 판매 가능한 수준의 UX, 성능, 확장성
6. AI 네이티브   → AI/LLM 서비스 모니터링을 1등 시민으로 설계
```

### 1.3 대상 사용자 (Persona)

| Persona | 역할 | 주요 관심사 | 사용 빈도 |
|---------|------|-----------|---------|
| **SRE / DevOps** | 인프라 운영, 장애 대응 | 실시간 알림, 서버 상태, 에러율 | 상시 (24/7) |
| **AI Engineer** | LLM/RAG 서비스 개발·운영 | TTFT, TPS, GPU 사용률, 모델 성능 | 매일 |
| **아키텍트 / 컨설턴트** | 시스템 점검, 진단 보고서 | IT-AI 교차분석, 종합 대시보드, 보고서 | 주기적 (주/월) |
| **경영진 / PM** | 서비스 상태 개요, SLO 준수 | Executive 대시보드, 비용, 트렌드 | 주간/월간 |
| **고객 (멀티테넌트)** | 자기 프로젝트 모니터링 | 내 서비스 상태, 알림, 보고서 | 수시 |

## 2. 경쟁 APM 벤치마크 분석

### 2.1 주요 APM 솔루션 UI 특성 비교

| 기능 | Datadog | New Relic | Dynatrace | Grafana | **우리 설계** |
|------|---------|-----------|-----------|---------|------------|
| 통합 네비게이션 | 좌측 사이드바 + 상단 검색 | 좌측 사이드바 + 글로벌 서치 | 좌측 사이드바 + Smartscape | 좌측 사이드바 + 패널 기반 | **좌측 2단 사이드바 + Command Palette** |
| 토폴로지 맵 | Service Map | Distributed Tracing Map | Smartscape 3D | Node Graph | **서비스 토폴로지 맵 (2D/계층)** |
| 프로젝트/조직 | Organization > Team | Account > Sub-account | Environment > Host Group | Organization > Folder | **Organization > Project > Host Group** |
| AI/LLM 지원 | LLM Observability (2024~) | AI Monitoring | Davis AI (AIOps) | ML 플러그인 | **AI 네이티브 1등 시민** |
| 커스텀 대시보드 | 위젯 드래그&드롭 | 위젯 편집기 | 타일 기반 | 패널 + 변수 | **위젯 드래그&드롭 + 템플릿** |
| 검색 | 통합 검색 + 태그 필터 | NRQL 쿼리 | DQL 쿼리 | Explore (PromQL) | **통합 검색 + 쿼리 빌더 + Command Palette** |
| 알림 | Monitor + Incident | Alert Condition + Workflow | Problem + Davis | Alert Rule + Contact Point | **Alert Policy + Incident + Runbook** |
| 에이전트 관리 | Fleet Automation | Guided Install | OneAgent 자동 | Alloy (Agent) | **AITOP Agent Fleet Console** |

### 2.2 벤치마크에서 가져올 Best Practice

```
[Datadog에서 배울 점]
├── 통합 좌측 네비게이션 — 모든 기능을 한 곳에서 접근
├── Service Map — 서비스 간 의존관계와 상태를 한눈에
├── Tag-based 필터링 — 모든 곳에서 일관된 태그 필터
└── Notebook — 조사 과정을 기록하고 공유

[New Relic에서 배울 점]
├── NRQL 쿼리 — 강력한 데이터 탐색 쿼리 언어
├── Workload — 관련 서비스를 논리적으로 그룹화
├── AI Monitoring — LLM 호출별 토큰/비용/성능 추적
└── SLI/SLO 관리 — 서비스 수준 목표 대시보드

[Dynatrace에서 배울 점]
├── Smartscape — 인프라↔서비스↔프로세스 자동 토폴로지
├── Davis AI — AI 기반 근본 원인 분석 (RCA)
├── OneAgent — 단일 에이전트로 전체 스택 자동 계측
└── Problem 카드 — 관련 이벤트를 하나의 문제로 자동 그룹화

[Grafana에서 배울 점]
├── 패널 + 변수 시스템 — 유연한 대시보드 커스터마이징
├── Explore — 데이터 소스별 자유 탐색
├── Alert Manager — 계층적 알림 라우팅
└── 플러그인 생태계 — 확장 가능한 시각화
```

### 6.4 XLog/HeatMap 강화 설계

> **상세 설계**: [XLOG_DASHBOARD_REDESIGN.md](./XLOG_DASHBOARD_REDESIGN.md) §16~19 참조
> **참고 솔루션**: Scouter XLog, Jennifer Transaction View, WhaTap HeatMap

#### 6.4.1 시간 범위 컨트롤 공통 컴포넌트

두 가지 컨텍스트에 맞게 시간 범위 컨트롤을 분리 설계합니다.

**① 대시보드 임베드 — `TimeRangeArrows`**

```
┌──────────────────────────────────────────────────────┐
│  [←]  [→]   2026-03-23 16:00 ~ 16:15 (15분)  [↑] [↓] │
└──────────────────────────────────────────────────────┘
```

| 버튼 | 동작 |
|------|------|
| `←` | 시간 축 과거 이동 (범위 너비만큼 shift) |
| `→` | 시간 축 미래 이동 (LIVE 모드 자동 해제) |
| `↑` | 시간 범위 축소 — 줌 인 (범위 절반, 중앙 고정) |
| `↓` | 시간 범위 확대 — 줌 아웃 (범위 2배, 중앙 고정) |

**② 상세 뷰 — `TimeRangePicker`**

```
┌──────────────────────────────────────────────────────────────────┐
│  [5분] [15분] [1시간] [6시간] [1일] [커스텀]                       │
│  시작 [2026-03-23] [16:00:00]   종료 [2026-03-23] [16:15:00]     │
│  [←] [→] ⏸ PAUSED  16:00 ~ 16:15  [↑] [↓]         [🔴 LIVE]  │
└──────────────────────────────────────────────────────────────────┘
```

#### 6.4.2 서버 멀티 셀렉터 — `ServerMultiSelector`

```
┌───────────────────────────────────────────────────────────────────┐
│ 대상 서버: [prod-app-01 ✓] [prod-app-02 ✓] [▼ 서버 선택]           │
│            ↓ 드롭다운                                               │
│  ☑ prod-app-01  ☑ prod-app-02  ☐ prod-app-03  ☐ prod-gpu-01     │
│                                     [전체 선택] [초기화] [적용]     │
└───────────────────────────────────────────────────────────────────┘
```

- 서버 체크박스 다중 선택 (최대 10개)
- 선택 서버별 다른 색상 점으로 XLog 산점도에 구분 표시
- 범례 자동 생성 + 클릭으로 서버 표시/숨김 토글

#### 6.4.3 XLog 트랜잭션 뷰 강화

- **날짜/시간 범위 지정 조회**: 상세 뷰에서 날짜 피커로 과거 구간 직접 입력
- **복수 서버 동시 조회**: 서버별 색상 구분 + 통합 산점도
- **에러 트랜잭션**: 빨간 점 `●` (z-index 최상위, 정상 점에 가려지지 않음)
- **대시보드 vs 상세 뷰**:
  - 대시보드: `TimeRangeArrows`만 표시 (화살표 4개)
  - 상세 뷰: `TimeRangePicker` 풀 UI + 서버 멀티 셀렉터

#### 6.4.4 히트맵 분리 패널 (WhaTap 스타일)

```
 응답시간 분포 밀도 히트맵
 ─────────────────────────────────────────────────────
  3s   ┤      ░░      ░░            ░░▒▒
  1s   ┤  ░░  ▓▓  ▒▒  ▓▓  ░░  ▒▒  ▓▓████
 200ms ┤  ██  ██  ██  ██  ▓▓  ██  ████████  ●● (에러)
  0ms  ├──────────────────────────────────→ 시간
       16:00        16:05        16:10        16:15

 ░ 1~10건  ▒ 11~50건  ▓ 51~200건  █ 201건+  ● 에러 집중
```

- **현재**: XLog와 HeatMap이 사이드바 탭으로 전환 (동시 표시 없음)
- **강화**: 별도 HeatMap 패널 뷰 추가 + XLog/HeatMap 분할 화면 모드 지원
- **드래그 선택**: 히트맵 영역 마우스 드래그 → 해당 구간 트랜잭션 목록 필터링
- **에러 오버레이**: 에러 비율 ≥ 10% 셀에 빨간 점 `●` 자동 표시
- **시간 탐색**: 히트맵에서도 `TimeRangeArrows` (←→↑↓) 동작

## 8.7 미들웨어 대시보드

> **목표**: 언어/프레임워크별로 특화된 미들웨어 전용 대시보드를 자동 생성하여, 개발자와 운영자가 즉시 런타임 상태를 파악할 수 있도록 한다.

### 8.7.1 언어 자동 감지 후 패널 자동 표시

에이전트 Heartbeat의 `runtime_language` 필드와 OTel `telemetry.sdk.language` 속성을 기반으로 해당 언어에 맞는 미들웨어 패널을 자동으로 활성화한다.

```
감지 흐름:
  Heartbeat / OTel 데이터 수신
    → runtime_language: "java" / "dotnet" / "nodejs" / "python" / "go"
    → 해당 언어의 미들웨어 패널 세트 자동 활성화
    → 미지원 항목은 "N/A (언어 미지원)" 라벨로 그레이아웃 처리

패널 활성화 규칙:
  Java    → Thread Pool + Connection Pool (HikariCP/DBCP) + Session + Request Queue
  .NET    → Kestrel Connection + Thread Pool Queue + GC/Heap + EF Core Pool
  Node.js → Event Loop Lag/Utilization + Active Connections + DB Pool (pg/mongoose)
  Python  → Worker Pool (Gunicorn) + Request Queue + SQLAlchemy Pool
  Go      → Goroutine Count + database/sql Pool
  공통    → HTTP Duration + 상태코드 분포 + TPS + 에러율
```

## 부록 C: 데모 계정 (개발/테스트용)

프론트엔드 개발 및 데모 시 백엔드 없이 동작하는 내장 데모 계정이 제공된다.
`demoLogin()` 함수를 통해 즉시 로그인이 가능하며, 각 역할별 권한 차이를 테스트할 수 있다.

| 역할 | 이메일 | 비밀번호 | 접근 범위 |
|------|--------|---------|---------|
| **Administrator** | `admin@aitop.io` | `admin` | 전체 기능 (설정, 사용자 관리, 삭제 포함) |
| **SRE / DevOps** | `sre@aitop.io` | `sre` | 프로젝트/인프라/서비스/AI/알림/에이전트/진단 읽기+쓰기, 설정 읽기 |
| **AI Engineer** | `ai@aitop.io` | `ai` | AI 서비스 읽기+쓰기, 나머지 읽기 전용 |
| **Viewer** | `viewer@aitop.io` | `viewer` | 전체 읽기 전용 (설정/에이전트 접근 불가) |

**역할 계층** (상위가 하위 권한을 포함):
```
admin (100) > sre (80) > ai_engineer (60) > viewer (10)
```

**권한 체크 컴포넌트**:
- `<RequireRole minRole="sre">` — 최소 역할 수준 이상만 렌더링
- `<RequirePermission resource="alerts" action="write">` — 특정 리소스/액션 권한 체크
- `useAuthStore().canAccess('ai', 'write')` — 프로그래밍 방식 권한 체크

> **주의**: 데모 계정은 개발/테스트 전용이다. 프로덕션 배포 시 반드시 실제 백엔드 인증(`/api/v1/auth/login`)으로 전환하고, `demoLogin()`을 비활성화해야 한다.

## 16. Virtual Thread 가젯 (Phase 39)

> **전제 조건**: JDK 21+ 감지 시 Java 미들웨어 대시보드에 Virtual Thread 가젯 섹션이 자동으로 표시됩니다. JDK 17 이하에서는 숨김 처리됩니다.

### 16.1 가젯 배치 — Java 미들웨어 대시보드 확장

기존 `/java` 미들웨어 상세 화면 하단에 **"Virtual Thread 모니터링 (JDK 21+)"** 섹션을 추가합니다.

```
┌─────────────────────────────────────────────────────────────────────┐
│  Java 미들웨어 대시보드  ·  my-spring-app (192.168.1.10)             │
│  JDK 21.0.2  [🟢 Virtual Thread 모니터링 활성]                      │
├──────────────────────────┬──────────────────────────────────────────┤
│  [기존 가젯들: TPS/RT/...]│                                          │
├──────────────────────────┴──────────────────────────────────────────┤
│  ▼ Virtual Thread 모니터링 (JDK 21+)                                 │
│  ┌─────────────────────┐  ┌─────────────────────────────────────┐   │
│  │ Carrier Pool 게이지  │  │  Virtual Thread 카운트 (실시간)     │   │
│  │                     │  │                                     │   │
│  │  ██████░░░░  6/8    │  │  활성: 12,483  대기: 3,201         │   │
```

### 16.2 가젯 1 — Carrier Pool 게이지

| 항목 | 상세 |
|------|------|
| **목적** | ForkJoinPool(Carrier Thread Pool) 점유율 시각화 |
| **표시 항목** | active / parallelism (예: 6/8), 점유율 % |
| **색상 임계** | < 70% 녹색, 70~90% 노랑, > 90% 빨강 |
| **데이터 소스** | `jvm.carrier_pool.active`, `jvm.carrier_pool.parallelism` |
| **갱신 주기** | 10초 |

### 16.3 가젯 2 — Virtual Thread 카운트 (실시간)

| 항목 | 상세 |
|------|------|
| **목적** | 현재 활성·대기 중인 Virtual Thread 수 및 생성 속도 |
| **표시 항목** | 활성(Active), 대기(Waiting/Parked), 최근 5분 생성 수 |
| **데이터 소스** | `jvm.virtual_thread.count{state="active|waiting"}` |
| **갱신 주기** | 5초 (SSE 실시간) |
| **특이사항** | Thread 수가 10만 초과 시 배지로 강조 ("10만+ 🚀") |

### 16.4 가젯 3 — Pinning 경고 (빨간색 알림)

| 항목 | 상세 |
|------|------|
| **목적** | synchronized 블록 Pinning 감지 및 원인 스택 표시 |
| **표시 항목** | 최근 1분 Pinning 이벤트 수, p99 지속 시간(ms), 상위 발생 메소드 Top-3 |
| **색상** | Pinning 없음: 녹색 배지, 1~9건: 노랑 ⚠️, 10건 이상: 빨강 🔴 |
| **데이터 소스** | `jvm.virtual_thread.pinned.count`, `jvm.virtual_thread.pinned.duration_ms` |
| **드릴다운** | 클릭 시 Pinning 스택 트레이스 목록 슬라이드패널 (JFR stackTrace 기반) |

### 16.5 가젯 4 — 스케줄링 지연 차트

| 항목 | 상세 |
|------|------|
| **목적** | Carrier Pool 포화로 인한 Virtual Thread 스케줄링 실패 추이 |
| **차트 타입** | Bar chart (1분 버킷, 최근 30분) |
| **표시 항목** | `jdk.VirtualThreadSubmitFailed` 이벤트 수/분 |
| **색상** | 0건: 녹색, 1~4건: 노랑, 5건 이상: 빨강 |
| **데이터 소스** | `jvm.virtual_thread.submit_failed` |
| **알림 연동** | 5건/분 초과 시 인시던트 자동 생성 |

### 16.6 JDK 버전 감지 배지

미들웨어 대시보드 헤더에 JDK 버전 배지를 추가합니다.

| JDK 버전 | 배지 | Virtual Thread 섹션 |
|---------|------|-------------------|
| JDK 17 이하 | `JDK 17` (회색) | 숨김 |
| JDK 21 이상 | `JDK 21 🟢` (녹색) | 자동 표시 |
| JDK 21+ + Pinning 감지 | `JDK 21 ⚠️` (노랑) | 자동 표시 + 경고 |
