## 1. 시작하기

### 1.1 로그인 및 초기 설정

브라우저에서 AITOP URL에 접속하면 로그인 화면이 표시됩니다.

**이메일/비밀번호 로그인**:
1. 이메일 주소와 비밀번호 입력
2. **로그인** 클릭
3. 최초 로그인 시 비밀번호 변경 안내가 표시됩니다

**SSO 로그인** (OIDC/SAML 설정된 경우):
1. **SSO로 로그인** 버튼 클릭
2. 회사 IdP(Okta, Azure AD, Google 등) 인증 후 자동 리디렉션

> **MFA(다단계 인증)**: 관리자가 활성화한 경우, 로그인 후 OTP 입력 화면이 추가로 표시됩니다.

**프로필 및 언어 설정**:
1. 우측 상단 사용자 아이콘 클릭
2. **프로필 설정** 선택
3. 언어(한국어/영어/일본어), 타임존, 알림 수신 방법 설정

### 1.2 프로젝트 생성

AITOP은 **프로젝트** 단위로 모니터링 대상을 격리합니다. 서비스 또는 팀 단위로 프로젝트를 분리하는 것을 권장합니다.

1. 좌측 상단 **프로젝트 선택기** 클릭
2. **새 프로젝트 만들기** 클릭
3. 프로젝트 이름, 설명 입력
4. **생성** 클릭
5. 발급된 **프로젝트 토큰** 복사 → Agent `agent.yaml`의 `project_token`에 입력

## 2. 대시보드

### 2.1 서비스 맵 (Service Map)

서비스 간 의존성과 트래픽 흐름을 실시간으로 시각화합니다.

**접속**: 좌측 메뉴 → **대시보드 → 서비스 맵**

**읽는 법**:
- **노드(원형)**: 각 서비스. 색상으로 상태 표시
  - 초록: 정상 (Apdex ≥ 0.9)
  - 노란색: 경고 (Apdex 0.7~0.9)
  - 빨간색: 장애 (Apdex < 0.7)
  - 회색: 데이터 없음
- **엣지(화살표)**: 서비스 간 호출. 두께는 TPS 비례, 색상은 오류율
- **노드 위 숫자**: TPS / 응답 시간 / 오류율

**상호작용**:
- 노드 클릭: 해당 서비스 상세 대시보드로 이동
- 엣지 클릭: 두 서비스 간 트레이스 목록 팝업
- 드래그: 지도 이동
- 스크롤: 확대/축소
- 우클릭: 서비스 필터, 트레이스 검색, 진단 실행 컨텍스트 메뉴

**레이아웃**:
- 상단 헤더의 **레이아웃** 버튼으로 계층형(Hierarchical) / 방사형(Radial) / 힘-방향(Force-directed) 전환

### 2.2 인프라 개요

모니터링 대상 서버의 리소스 현황을 한눈에 봅니다.

**접속**: **대시보드 → 인프라 개요**

표시 정보:
- 총 호스트 수, 활성/비활성 수
- 호스트별 CPU/메모리/디스크/네트워크 사용률 히트맵
- Top 10 CPU 사용 호스트, Top 10 메모리 사용 호스트
- GPU 장착 서버 VRAM 사용률 현황

### 2.3 커스텀 대시보드

원하는 지표를 조합하여 나만의 대시보드를 만듭니다.

**새 대시보드 생성**:
1. **대시보드 → 커스텀** 접속
2. **+ 새 대시보드** 클릭
3. 대시보드 이름, 설명 입력
4. **위젯 추가** 버튼 클릭:
   - 지표 유형 선택 (메트릭/로그/트레이스 카운트)
   - PromQL 쿼리 입력 또는 빌더로 구성
   - 시각화 유형 선택 (라인 차트/바 차트/게이지/히트맵/테이블)
   - 임계값 설정 (색상 경고선)
5. 위젯 드래그앤드롭으로 레이아웃 조정
6. **저장** 클릭

**대시보드 공유**:
- **공유** 버튼 → 읽기 전용 링크 생성
- 팀 공유: 팀/역할을 대상으로 공유 설정
- Marketplace에 공개: **Marketplace에 게시** 선택

## 3. XLog 및 히트맵

XLog/히트맵은 요청별 응답 시간 분포를 실시간으로 시각화하여 이상을 즉시 감지하는 핵심 화면입니다.

### 3.1 히트맵 읽는 법

**접속**: **APM → XLog / 히트맵**

```
응답  |  ■ ■ ■ ■ ■        ← 높은 농도 = 많은 요청 수
시간  |  ■ ■ ■ ■ ■ ■ ■ ■
(ms) |  ■ ■ ■ ■ ■ ■ ■ ■ ■
     |  ■ ■ ■
     └──────────────────── 시간 축 (좌→우 = 과거→현재)
```

- **X축**: 시간 (실시간으로 좌에서 우로 흐름)
- **Y축**: 응답 시간 (ms). 위로 갈수록 느린 요청
- **색상 농도**: 동일 시간대에 동일 응답 시간을 가진 요청 수 (연노란색 → 진빨간색)
- **이상 패턴 해석**:
  - 히트맵이 위로 올라감: 응답 시간 증가
  - 특정 시간에 빨간 점: 일시적 급격한 지연
  - 두 줄로 갈림: 두 종류의 요청이 혼재 (캐시 히트/미스 등)

### 3.2 XLog 테이블

히트맵 아래에는 개별 요청 목록이 표시됩니다.

| 컬럼 | 설명 |
|------|------|
| 시각 | 요청 수신 시각 |
| 서비스 | 요청을 받은 서비스명 |
| URL/메서드 | 요청 경로 및 HTTP 메서드 |
| 응답 시간 | 총 처리 시간 (ms) |
| 상태 | HTTP 상태 코드 |
| 트레이스 ID | 클릭 시 상세 트레이스로 이동 |
| CPU 시간 | 서버 CPU 점유 시간 |

**클릭하면**: 해당 요청의 분산 트레이스 상세 화면으로 이동합니다.

### 3.3 요청 필터링

히트맵에서 원하는 영역을 드래그하면 해당 시간대/응답 시간 범위의 요청만 필터링합니다.

추가 필터:
- **서비스 선택**: 특정 서비스만 보기
- **URL 패턴**: `/api/v1/chat*` 등 glob 패턴
- **상태 코드**: 오류만 보기 (`≥ 400`)
- **응답 시간**: 특정 범위 이상만 (`> 500ms`)
- **사용자 ID**: 특정 사용자의 요청만
- **트레이스 속성**: 임의 Span 속성 기반 필터

## 4. 분산 트레이스

### 4.1 트레이스 검색

**접속**: **APM → 트레이스 검색**

검색 조건:
- **트레이스 ID**: 직접 ID 입력
- **서비스**: 요청이 통과한 서비스 선택
- **URL**: `/chat/completions` 등
- **기간**: 최근 1시간 ~ 7일
- **응답 시간**: 특정 ms 이상
- **오류 여부**: 오류 포함 트레이스만
- **속성 검색**: `llm.model=gpt-4o`, `user.id=1234` 등

검색 결과는 응답 시간 기준으로 정렬되며, 상위 항목부터 느린 요청을 확인합니다.

### 4.2 워터폴(Waterfall) 분석

트레이스를 클릭하면 각 구간(Span)의 실행 시간을 시각적으로 보여주는 워터폴 다이어그램이 열립니다.

```
Span 이름                    |  0ms   100ms  200ms  300ms  400ms
─────────────────────────────────────────────────────────────────
API Gateway                  │████████████████████████████████ 400ms
  ├─ Auth Check              │███ 30ms
  ├─ Guardrail Check         │   ████ 45ms
  ├─ LangChain Agent         │       █████████████████ 290ms
  │   ├─ Vector Search       │       ████████ 90ms
  │   │   └─ Milvus Query    │       ████ 40ms
  │   └─ LLM API Call        │               █████████ 120ms
  │       └─ OpenAI GPT-4o   │               █████████ 120ms
  └─ Response Format         │                         ███ 35ms
```

**Span 클릭 시 상세 정보**:
- 시작 시각, 종료 시각, 지속 시간
- 서비스명, 호스트, 프로세스 ID
- 모든 Span 속성 (HTTP URL, DB 쿼리, LLM 모델명, 입출력 토큰 수 등)
- 오류 발생 시 스택 트레이스

**AI 서비스 전용 Span 속성**:

| 속성 | 설명 |
|------|------|
| `llm.model` | 사용된 LLM 모델명 |
| `llm.input_tokens` | 입력 토큰 수 |
| `llm.output_tokens` | 출력 토큰 수 |
| `llm.ttft_ms` | 첫 토큰까지 대기 시간 |
| `llm.total_cost_usd` | 해당 호출 비용 |
| `rag.retrieved_docs` | 검색된 문서 수 |
| `rag.relevance_score` | 검색 관련도 |
| `guardrail.blocked` | 가드레일 차단 여부 |

### 4.3 트레이스 비교

두 트레이스를 나란히 비교하여 성능 차이를 분석합니다.

1. 트레이스 목록에서 비교할 두 트레이스 체크박스 선택
2. **비교 보기** 버튼 클릭
3. 좌우로 나란히 워터폴이 표시됨
4. 시간 차이가 큰 Span이 강조 표시됨

## 5. AI 서비스 모니터링

### 5.1 LLM 서비스 현황

**접속**: **AI 서비스 → LLM 현황**

**핵심 지표**:

| 지표 | 설명 | 정상 범위 (예시) |
|------|------|----------------|
| TTFT (Time To First Token) | 첫 토큰 응답까지 걸린 시간 | < 500ms |
| TPS (Tokens Per Second) | 초당 생성 토큰 수 | 모델별 상이 |
| ms/token | 토큰 1개 생성 소요 시간 | < 20ms |
| 입력/출력 토큰 비율 | 프롬프트 대비 응답 토큰 비율 | 서비스별 정의 |
| 오류율 | API 호출 실패 비율 | < 1% |
| 모델별 요청 분포 | 어떤 모델을 얼마나 쓰는지 | — |

**TTFT 분포 히스토그램**: 요청의 응답 시간 분포를 확인합니다. 꼬리 부분(P99)이 높으면 일부 사용자가 매우 느린 경험을 하고 있다는 신호입니다.

**모델 비교**: 여러 모델을 동시에 사용하는 경우, 모델별 TTFT/비용/오류율을 비교합니다.

### 5.2 GPU 모니터링

      ↑              ↑               ↑               ↑              ↑
   레이턴시        레이턴시         레이턴시         레이턴시       레이턴시
   추적됨          추적됨           추적됨           추적됨         추적됨

### 5.4 가드레일(Guardrail)

**차단 이벤트 목록**: 각 차단 이벤트를 클릭하면 차단 이유, 관련 트레이스, 원본 요청(마스킹 처리)을 확인할 수 있습니다.

## 6. 프로파일링

### 6.1 FlameGraph 보는 법

**접속**: **프로파일링 → FlameGraph**

FlameGraph는 CPU 시간이 어떤 함수에서 얼마나 사용되는지를 시각적으로 표현합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│  handleRequest (100%)                                           │
├───────────────────────────────────────┬─────────────────────────┤
│  processLLMCall (60%)                 │  processDB (35%)        │
├──────────────────┬────────────────────┤────────────┬────────────┤
│  tokenize (25%)  │  callOpenAI (35%)  │ query(20%) │ parse(15%) │
└──────────────────┴────────────────────┴────────────┴────────────┘
```

- **너비**: CPU 시간 비율. 넓을수록 많은 CPU 시간 사용
- **높이**: 콜스택 깊이. 아래가 호출자(Caller), 위가 피호출자(Callee)
- **색상**: 언어별 구분 (파란색=Java, 주황색=Python, 초록색=Go 등)
- **가장 넓은 평평한 스택이 병목**: FlameGraph 상단에서 넓고 평평한 함수가 병목

**조작 방법**:
- 클릭: 해당 함수를 루트로 확대
- 우클릭: 함수 소스 코드로 이동 (코드 링크 연동 시)
- 검색 (`Ctrl+F`): 특정 함수명 하이라이트
- Reset 버튼: 전체 보기로 복귀

### 6.2 CPU vs Off-CPU 분석

| 유형 | 의미 | 병목 원인 |
|------|------|----------|
| CPU (On-CPU) | CPU 연산에 실제로 사용된 시간 | 연산 집약적 코드, 비효율 알고리즘 |
| Off-CPU | 블록 대기 시간 (I/O, 락, 슬립) | DB 쿼리, 파일 읽기, 락 경합 |

화면 상단의 **CPU / Off-CPU / Memory** 탭으로 전환합니다.

> **팁**: 응답 시간은 느린데 CPU FlameGraph에서 병목이 안 보인다면 Off-CPU를 확인하세요. DB 쿼리나 락 대기가 원인일 가능성이 높습니다.

### 6.3 프로파일 비교(Diff)

배포 전후 또는 두 시점의 프로파일을 비교하여 성능 변화를 파악합니다.

1. **프로파일링 → 비교(Diff)** 접속
2. 기준(Base) 프로파일 선택 (배포 이전 스냅샷)
3. 비교(Compare) 프로파일 선택 (배포 이후 스냅샷)
4. **비교 시작** 클릭
5. 빨간색 = 배포 후 느려진 함수, 파란색 = 빨라진 함수

## 7. AI Copilot

### 7.1 자연어 쿼리

**접속**: **Copilot** 또는 검색창에서 `/` 입력

자연어로 원하는 정보를 물어보면 PromQL로 변환하여 결과를 보여줍니다.

**예시 질문**:

```
"지난 1시간 동안 가장 느린 서비스 3개 알려줘"
→ PromQL: topk(3, avg_over_time(http_server_duration_ms[1h]))

"오늘 GPU 사용률이 90%를 넘은 시간대가 언제야?"
→ PromQL: sum(aitop_gpu_utilization > 90) by (host)

"주문 서비스의 오류율 추세를 7일간 보여줘"
→ PromQL: rate(http_requests_total{service="order-service",status=~"5.."}[5m])
```

Copilot이 생성한 PromQL은 **편집 후 저장**하여 대시보드 위젯으로 추가할 수 있습니다.

### 7.2 장애 분석 요청

장애 발생 시 Copilot에 상황을 설명하면 관련 메트릭, 트레이스, 로그를 자동으로 조합하여 분석 결과를 제공합니다.

```
사용자: "오후 2시부터 채팅 서비스 응답 시간이 3배 늘었어. 원인 찾아줘"

Copilot 분석 결과:
1. 14:00에 chat-service P99 응답 시간 350ms → 1,100ms 급증
2. 동시간대 Vector DB(Milvus) 검색 레이턴시 300ms → 900ms 상승
3. Milvus 호스트 디스크 I/O saturation 89% 관찰
4. 원인 추정: Milvus 인덱싱 작업과 검색 쿼리 동시 실행으로 디스크 경합
5. 권고 조치: 인덱싱 스케줄을 비업무 시간으로 조정
```

### 7.3 Investigation Notebook

복잡한 장애 분석 과정을 여러 팀원과 협업하며 기록합니다.

**접속**: **Copilot → Notebook** 또는 인시던트에서 **Notebook 생성**

노트북은 다음으로 구성됩니다:
- **텍스트 셀**: 마크다운 형식의 분석 메모
- **쿼리 셀**: PromQL 쿼리 + 실행 결과 차트
- **트레이스 셀**: 특정 트레이스 임베드
- **Copilot 셀**: AI 분석 요청 + 결과

노트북은 팀과 공유하고, 인시던트 포스트모템에 첨부할 수 있습니다.

## 8. Fleet 관리

### 8.1 Agent 현황 조회

**접속**: **Fleet → Agents**

| 컬럼 | 설명 |
|------|------|
| 서버명 | Agent가 설치된 서버 호스트명 |
| IP | 서버 IP 주소 |
| 버전 | Agent 버전 (최신 버전과 다르면 노란색 강조) |

**상태별 의미**:

**Agent 상세 보기**: 클릭하면 해당 서버의 Collector 상태, 최근 오류, 버전 이력을 확인합니다.

### 8.2 원격 CLI

**접속**: **Fleet → Agents → 대상 서버 선택 → Remote CLI**

브라우저에서 해당 서버에 직접 터미널 명령을 실행합니다.

```bash
# 접속 후 일반 터미널처럼 사용 가능
$ ls -la /var/log/
$ top -b -n 1
$ ps aux | grep java

# AITOP Agent 명령
$ aitop-agent status
$ aitop-agent collectors status
$ aitop-agent diagnose
```

**제한 사항**:
- 관리자 또는 SRE 역할만 사용 가능
- 모든 명령이 감사 로그에 기록됨
- 일부 위험 명령은 차단됨 (`rm -rf /`, `shutdown` 등)
- 세션 타임아웃: 유휴 10분, 최대 1시간

### 8.3 서버 그룹 관리

**접속**: **Fleet → Groups**

서버를 논리적 그룹으로 구성하여 대시보드 필터, 알림 정책, OTA 업데이트를 그룹 단위로 적용합니다.

**그룹 생성**:
1. **+ 그룹 추가** 클릭
2. 그룹 이름 입력 (예: `production-web`, `gpu-servers`)
3. 서버 추가 방법 선택:
   - 수동 선택: Agent 목록에서 선택
   - 자동 규칙: 호스트명 패턴, OS, IP 대역 기반 자동 분류
4. **저장**

## 9. 알림(Alert) 관리

### 9.1 알림 정책 생성

**접속**: **알림 → 정책 → + 새 정책**

**기본 설정**:
1. 정책 이름 및 설명 입력
2. 조건 설정:
   - **PromQL 조건**: `rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) > 0.05`
   - **임계값**: 위 조건이 `true`이면 발동
   - **지속 시간**: 5분 이상 지속 시 알림 (순간 스파이크 무시)
3. 심각도 선택: Critical / Warning / Info
4. 알림 채널 선택 (9.3절 참조)
5. **알림 그룹**: 유사 알림을 묶어서 발송 (노이즈 감소)
6. **억제 규칙**: 상위 알림 발생 시 하위 알림 억제

**빠른 알림 생성**: 대시보드 위젯에서 우클릭 → **이 지표로 알림 생성** 선택

### 9.2 인시던트 관리

알림이 발생하면 자동으로 인시던트가 생성됩니다.

**접속**: **알림 → 인시던트**

**인시던트 상세 화면**:
- 발생 시각, 지속 시간
- 영향 받은 서비스 및 서버
- 관련 메트릭 차트 (알림 조건 기준)
- 자동 수집된 관련 트레이스/로그
- Copilot AI 분석 결과 (자동 생성)
- 담당자 지정 및 상태 변경 (Open / Investigating / Resolved)
- 타임라인 (댓글, 조치 기록)

**인시던트 해결**:
1. 원인 파악 후 조치 완료
2. **해결 처리** 버튼 클릭
3. 해결 요약 입력 (포스트모템 자동 생성에 활용)

### 9.3 알림 채널 설정

**접속**: **설정 → 알림 채널 → + 채널 추가**

지원 채널:

| 채널 | 설정 내용 |
|------|----------|
| **이메일** | 수신자 이메일 주소 목록 |
| **Slack** | Webhook URL, 채널명 |
| **Teams** | Webhook URL |
| **PagerDuty** | Integration Key |
| **Webhook** | URL, HTTP 메서드, 헤더, 페이로드 템플릿 |
| **SMS** | 전화번호 (Twilio 연동 필요) |

**알림 라우팅**: 심각도에 따라 다른 채널로 발송합니다.
- Critical → PagerDuty + Slack #incident
- Warning → Slack #monitoring
- Info → 이메일 (일 1회 요약)

## 10. 진단 보고서

### 10.1 자동 진단 실행

AITOP은 86개 항목 (IT 55개 + AI 31개)을 자동으로 진단합니다.

**접속**: **진단 → 진단 실행**

**즉시 실행**:

3. **진단 시작** 클릭
4. 약 2~5분 후 결과 확인

**정기 실행 설정**:
- **설정 → 진단 스케줄** 에서 cron 표현식으로 자동화
- 결과를 이메일 또는 Slack으로 자동 발송

### 10.2 진단 항목 해석

진단 결과는 4가지 등급으로 분류됩니다:

| 등급 | 의미 |
|------|------|
| ✅ 정상 | 기준값 이내 |
| ⚠️ 주의 | 기준값 경계, 모니터링 필요 |
| 🔴 위험 | 즉각 조치 권고 |
| ❓ 확인 불가 | 데이터 부족 또는 접근 불가 |

**주요 IT 진단 항목** (55개 중 예시):
- JVM Heap 사용률 80% 초과
- DB 슬로우 쿼리 빈도 (임계값: 분당 5회 이상)
- 디스크 I/O 대기 시간 (임계값: 50ms 이상)
- TCP TIME_WAIT 연결 과다 (임계값: 10,000개 이상)
- 파일 디스크립터 사용률 (임계값: 80% 이상)

**주요 AI 진단 항목** (31개 중 예시):
- TTFT 이상 (P99 > 2초)
- GPU VRAM 누수 (실행 간 VRAM 미반환)
- 토큰 소모 급증 (일일 한도 80% 초과)
- 벡터 DB 인덱스 단편화 (fragmentation > 30%)
- 가드레일 차단율 이상 급증

각 항목 클릭 시 상세 설명, 관련 메트릭, 권고 조치 방법이 표시됩니다.

## 11. SLO 관리

### 11.1 SLO 생성

SLO(Service Level Objective)를 설정하여 서비스 품질 목표를 관리합니다.

**접속**: **SLO → + 새 SLO**

1. **SLO 이름** 입력 (예: `Chat API 가용성`)
2. **서비스** 선택
3. **SLI 유형** 선택:
   - **가용성**: 성공 요청 비율 (성공 기준: HTTP 2xx/3xx)
   - **레이턴시**: 특정 응답 시간 이하 비율 (예: P95 < 200ms)
   - **오류율**: 오류 요청 비율 (HTTP 5xx 기준)
   - **커스텀 PromQL**: 임의 SLI 표현식
4. **목표값** 입력 (예: 가용성 99.9%)
5. **윈도우** 선택: 롤링 28일 / 캘린더 월 / 분기
6. **저장**

### 11.2 Error Budget 확인

**접속**: **SLO → 대시보드**

**Error Budget이란**: SLO 목표를 달성하기 위해 허용된 오류/다운타임 비율입니다.
- 99.9% SLO = 0.1% Error Budget = 월 43.8분 허용

**화면 구성**:
- **남은 Error Budget 비율**: 100%에서 소진 비율을 차감
- **Burn Rate**: Error Budget 소진 속도 (1.0 = SLO 경계선 속도)
  - Burn Rate > 1 → 현재 속도로는 기간 내 Error Budget 초과 위험
  - Burn Rate > 14 → 1시간 내 월간 Error Budget 10% 소진 (즉각 대응)
- **SLO 이력**: 과거 달성 여부 캘린더 뷰

**Error Budget 알림**:
- Budget 50% 소진 → Warning 알림
- Budget 90% 소진 → Critical 알림
- Burn Rate 14x 이상 → 즉시 알림

**접속**: **설정 → 프로젝트**

- **기본 정보**: 프로젝트 이름, 설명, 아이콘 변경
- **토큰 관리**: 프로젝트 토큰 재발급, 만료 설정
- **데이터 보관**: 메트릭/트레이스 보관 기간 프로젝트별 설정
- **알림 기본값**: 프로젝트 기본 알림 채널 설정
- **White-label**: 로고, 색상 커스터마이징 (엔터프라이즈)

### 13.2 사용자 및 권한

**접속**: **설정 → 사용자**

| 역할 | 권한 |
|------|------|
| admin | 모든 기능, 사용자 관리, 설정 변경 |
| sre | 읽기 + Remote CLI + 알림 관리 + 진단 실행 |
| dev | 읽기 + 트레이스/로그/프로파일링 상세 |
| viewer | 대시보드/메트릭 조회만 가능 |

**사용자 초대**:
1. **사용자 초대** 클릭
2. 이메일 입력, 역할 선택
3. **초대 발송** → 이메일로 가입 링크 전송

**팀 구성**: 여러 사용자를 팀으로 묶어 공통 권한 및 알림 설정 적용 가능

### 13.3 SSO 설정

**접속**: **설정 → SSO**

1. SSO 유형 선택 (OIDC / SAML)
2. IdP 정보 입력 (Client ID, Secret, Metadata URL 등)
3. 역할 매핑 설정: IdP 그룹 → AITOP 역할 매핑
   - 예: `okta-group:sre-team` → AITOP `sre` 역할
4. **테스트 연결** 클릭 → 정상 확인 후 **저장**

## 14. FAQ

**Q: 서비스 맵에 일부 서비스가 보이지 않아요**
A: 해당 서비스에 SDK 계측(OTel)이 없거나 Agent가 설치되지 않은 경우입니다.
SDK를 추가하거나 Agent를 해당 서버에 설치하세요. (설치 가이드 9장 참조)

**Q: XLog 히트맵이 빈 화면으로 나와요**
A: 선택된 시간 범위에 데이터가 없거나, 서비스 필터가 잘못 선택되었을 수 있습니다.
시간 범위를 넓히거나 서비스 필터를 초기화해 보세요.

**Q: 알림이 오지 않아요**
A: 알림 채널 설정 (설정 → 알림 채널) 에서 **테스트 발송** 버튼을 클릭하여 채널이 정상인지 확인하세요.
그래도 오지 않는다면 Collection Server 로그에서 `notification` 관련 오류를 확인하세요.

**Q: TTFT 값이 너무 높게 나와요. 실제와 다른 것 같아요**
A: TTFT는 SDK 계측에서 스트리밍 첫 청크 수신 시각을 기록합니다.
애플리케이션 코드에서 `stream=True` 모드를 사용하고 있는지 확인하세요.
스트리밍을 사용하지 않는 경우 TTFT는 전체 응답 시간과 동일하게 측정됩니다.

**Q: GPU 메트릭이 수집되지 않아요**
A: `nvidia-smi` 명령이 Agent를 실행하는 사용자 계정에서 실행 가능한지 확인하세요.
DCGM을 사용하는 경우 DCGM Exporter URL 설정을 확인하세요.
자세한 내용은 설치 가이드 13.3절 트러블슈팅을 참조하세요.

**Q: FlameGraph에서 함수 이름이 `unknown` 또는 16진수로 표시돼요**
A: 심볼 분석 도구가 설치되지 않았거나 설정이 잘못되었습니다.
Java: `perf-map-agent`, Python: `py-spy` 설치 후 agent.yaml의 `symbol_resolvers` 설정을 확인하세요.

**Q: Remote CLI 접속이 안 돼요. "Permission denied"가 나와요**
A: 계정 역할이 `admin` 또는 `sre`여야 합니다.
설정 → 사용자에서 본인의 역할을 확인하거나 관리자에게 요청하세요.

**Q: SLO Error Budget이 갑자기 0%가 됐어요**
A: SLO 계산 윈도우(기간) 내에 대규모 장애가 발생했을 수 있습니다.
SLO 설정에서 **이력 보기**를 클릭하면 언제 어떤 장애로 Budget이 소진되었는지 확인할 수 있습니다.
Error Budget을 리셋할 수는 없으나, 다음 계산 기간 시작 시 자동으로 초기화됩니다.

**Q: Copilot이 엉뚱한 PromQL을 생성해요**
A: Copilot에게 더 구체적인 조건을 제공하세요.
예) "오늘" 대신 "오늘 오후 2시부터 3시 사이", "느린 서비스" 대신 "응답 시간 P99가 1초 이상인 서비스"

**Q: 대시보드가 너무 느려요**
A: 시간 범위를 단축하거나, 복잡한 PromQL 쿼리의 집계 간격을 늘려보세요.
관리자에게 Prometheus 쿼리 타임아웃 설정 증가를 요청하세요.
