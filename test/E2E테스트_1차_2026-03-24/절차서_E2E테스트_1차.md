## 4. AI 실행 절차 (Claude Code)

> AI가 먼저 실행하고, 결과를 `결과서_E2E테스트_1차_AI.md`에 기록합니다.

### Step A-1: Docker E2E 스택 기동

- **점검 목적**: E2E 테스트의 대상 환경을 준비한다. 통합테스트에서 사용한 동일 스택을 기동하되, Frontend가 실제 브라우저 접속 가능한 상태(localhost:3000)여야 한다. Playwright가 이 URL에 접속하여 UI를 조작하므로, 스택이 완전히 healthy 상태여야 테스트가 안정적으로 실행된다.

```bash
cd /c/workspace/aiservice-monitoring && docker compose -f docker-compose.e2e.yaml up -d --build
```

- **기대 결과**: 모든 서비스 healthy
- **판정 기준**: 헬스체크 PASS 시 진행

### Step A-2: Playwright E2E 시나리오 테스트

- **점검 목적**: 4가지 사용자 역할(SRE, AI Engineer, Consultant, Agent 관리자)의 핵심 워크플로우를 자동화된 브라우저에서 실행한다. 각 역할이 로그인 → 대시보드 확인 → 상세 페이지 드릴다운 → 데이터 조회까지의 전체 흐름을 완주할 수 있는지 검증한다. 또한 26개+ 라우트가 에러 없이 렌더링되는지, 404 페이지가 올바르게 표시되는지 확인한다. 이 테스트가 실패하면 실제 사용자가 해당 기능을 사용할 수 없다는 의미이다.

```bash
cd /c/workspace/aiservice-monitoring/frontend && npx playwright test --project=chromium 2>&1 | tee ../test/E2E테스트_1차_2026-03-24/logs/playwright-chromium-output.txt
```

대상 시나리오:

| # | Spec 파일 | 역할 | 검증하는 사용자 플로우 |
|---|----------|------|---------------------|
| 1 | `01-sre-incident-response.spec.ts` | SRE | Executive → Services → Trace 드릴다운, Alerts → 인시던트 → RCA |
| 2 | `02-ai-engineer-tuning.spec.ts` | AI Engineer | AI 개요 → AI 상세 → GPU 클러스터 → 진단 |
| 3 | `03-consultant-inspection.spec.ts` | Consultant | Projects → Agents → 진단 → SLO → 비용 |
| 4 | `04-agent-management.spec.ts` | Agent 관리자 | Fleet Console → Agent 목록 → Jobs → Plugins |
| 5 | `05-navigation-and-i18n.spec.ts` | 전체 | 26개 라우트 렌더링 + 로그인 + 404 |

- **기대 결과**: 9/9 시나리오 PASS
- **판정 기준**: FAIL 0건이면 PASS

### Step A-3: 접근성(a11y) 테스트

- **점검 목적**: WCAG 2.1 AA 기준을 충족하는지 axe-core로 자동 검사한다. 색상 대비 부족, aria-label 누락, 키보드 접근 불가 등의 접근성 위반은 시각/운동 장애 사용자가 솔루션을 사용할 수 없게 만든다. 상용 소프트웨어는 접근성 법규(ADA, EN 301 549) 준수가 필수이므로 위반 사항을 발견하고 수정해야 한다.

```bash
cd /c/workspace/aiservice-monitoring/frontend && npx playwright test --project=a11y 2>&1 | tee ../test/E2E테스트_1차_2026-03-24/logs/playwright-a11y-output.txt
```

- **기대 결과**: 위반 0건 또는 Minor만 존재
- **판정 기준**: Critical/Serious 위반 0건이면 PASS

### Step A-4: Visual Regression 테스트

- **점검 목적**: UI의 시각적 일관성을 스냅샷으로 검증한다. CSS 변경, 컴포넌트 리팩토링, 라이브러리 업데이트 후 의도하지 않은 레이아웃 깨짐이나 스타일 변경을 탐지한다. 첫 실행 시 기준 스냅샷을 생성하고, 이후 실행에서 픽셀 단위 차이를 비교한다. 예상치 못한 변경이 발견되면 의도된 변경인지 버그인지 사람이 판단해야 한다.

```bash
cd /c/workspace/aiservice-monitoring/frontend && npx playwright test --project=visual 2>&1 | tee ../test/E2E테스트_1차_2026-03-24/logs/playwright-visual-output.txt
```

- **기대 결과**: 기준 스냅샷 생성 또는 차이 0건
- **판정 기준**: 예상치 못한 시각적 변경 0건이면 PASS (첫 실행 시 기준 스냅샷 생성으로 자동 PASS)

### Step A-5: Go 통합 E2E 테스트

- **점검 목적**: Backend 컴포넌트들이 Docker 환경에서 실제로 연동되는지 Go 코드 레벨에서 검증한다. Validation Gateway가 파이프라인에서 올바르게 payload를 필터링하는지, EventBus가 SSE로 이벤트를 전파하는지, Storage Backend가 MinIO에 정상 저장하는지, JWT 멀티롤 인증이 실제 HTTP 레벨에서 동작하는지 확인한다. Playwright가 Frontend를 검증하듯, 이 테스트는 Backend의 E2E를 검증한다.

```bash
cd /c/workspace/aiservice-monitoring/agent && go test -v -count=1 ./test/ 2>&1 | tee ../test/E2E테스트_1차_2026-03-24/logs/go-e2e-output.txt
```

- **기대 결과**: 27+ 테스트 전체 PASS
- **판정 기준**: FAIL 0건이면 PASS

### Step A-6: 트레이스 연속성 검증

- **점검 목적**: W3C TraceContext가 5계층(Frontend → Collection Server → OTel Collector → Tempo → Prometheus)을 정상 전파하는지 확인한다. 트레이스가 중간에 끊어지면 장애 발생 시 근본 원인 추적이 불가능해지며, AITOP의 핵심 가치인 "정확한 진단"이 무력화된다. Baggage 아이템, 메트릭↔로그 상관관계도 함께 검증한다.

```bash
bash scripts/e2e/trace-continuity.sh 2>&1 | tee test/E2E테스트_1차_2026-03-24/logs/trace-continuity-output.txt
```

- **기대 결과**: 전파 체인 정상
- **판정 기준**: 끊김 0건이면 PASS

### Step A-7: 보안 감사

- **점검 목적**: OWASP Top 10 취약점(인젝션, XSS, 인증 우회 등), PII 마스킹 정상 동작, mTLS 설정을 검증한다. 모니터링 솔루션은 민감한 인프라 데이터를 다루므로, 보안 취약점은 고객 데이터 유출로 직결된다. 특히 API 인증 없이 접근 가능한 엔드포인트, 응답에 포함된 비마스킹 PII, 안전하지 않은 헤더를 탐지한다.

```bash
bash scripts/e2e/security-audit.sh 2>&1 | tee test/E2E테스트_1차_2026-03-24/logs/security-audit-output.txt
```

- **기대 결과**: Critical 취약점 0건
- **판정 기준**: Critical 0건이면 PASS, Medium 이하는 기록 후 CONDITIONAL PASS

### Step A-8: 부하 테스트 (Locust)

- **점검 목적**: 실제 운영 환경과 유사한 부하(200명 동시 사용자, 1000 RPS 목표)에서 시스템이 안정적으로 응답하는지 측정한다. P95 응답 시간이 2000ms를 초과하면 사용자가 "느리다"고 체감하고, 실패율이 1%를 초과하면 모니터링 데이터 유실이 발생한다. 4가지 시나리오(API 조회 60%, Heartbeat 20%, Agent 등록 10%, 수집 트리거 10%)로 실제 사용 패턴을 시뮬레이션한다.

```bash
cd /c/workspace/aiservice-monitoring && locust -f locust/locustfile.py --headless --users 200 --spawn-rate 20 --run-time 10m --host http://localhost:8080 2>&1 | tee test/E2E테스트_1차_2026-03-24/logs/locust-output.txt
```

SLO 기준:

| 지표 | 목표 | 판정 |
|------|------|------|
| P50 응답 시간 | < 500ms | |
| P95 응답 시간 | < 2000ms | **필수** |
| P99 응답 시간 | < 5000ms | |
| 실패율 | < 1% | **필수** |
| 목표 RPS | ~1000 req/s | |

- **기대 결과**: P95 < 2000ms, 실패율 < 1%
- **판정 기준**: 필수 2개 지표 충족 시 PASS

### Step A-9: AI-L4 성능 분석

- **점검 목적**: 부하 테스트 결과와 별개로, 코드 레벨에서 성능 병목이 될 수 있는 패턴을 정적으로 탐지한다. N+1 쿼리(for 루프 안에서 DB 호출), goroutine 누수 위험(close 안 된 채널), Frontend re-render 과다(의존성 없는 useEffect) 등은 테스트에서는 발견하기 어렵지만 운영 환경에서 점진적으로 성능을 저하시킨다.

- Backend: N+1 쿼리 패턴, 닫히지 않은 channel/goroutine
- Frontend: 불필요한 re-render, 대형 번들 import

- **판정 기준**: 위험 패턴 건수 기록 (정보 제공)

### Step A-10: AI-L5 문서 ↔ 코드 일관성 분석

- **점검 목적**: DOCS/ 문서에 기술된 아키텍처, API 스펙, 설정값이 실제 코드와 일치하는지 확인한다. 문서가 오래되어 현실과 다르면, 새로운 개발자나 고객이 잘못된 정보를 기반으로 작업하게 된다. 특히 ARCHITECTURE.md의 포트 번호, METRICS_DESIGN.md의 메트릭 이름, AGENT_DESIGN.md의 Collector 목록이 실제 코드와 일치하는지 대조한다.

- ARCHITECTURE.md ↔ docker-compose, Makefile 포트/서비스
- METRICS_DESIGN.md ↔ Prometheus 메트릭 이름
- AGENT_DESIGN.md ↔ agent/internal/collector/ 디렉토리

- **판정 기준**: 불일치 건수 기록 (정보 제공)

### Step A-11: Docker 스택 정리 + 리포트 수집

- **점검 목적**: 테스트 완료 후 리소스를 정리하고, Playwright HTML 리포트를 보존한다.

- **판정 기준**: 에러 없이 완료 시 PASS

## 5. 수동 검증 절차 (사용자)

> AI 결과를 참조하면서 수동으로 재검증합니다.
> 결과를 `결과서_E2E테스트_1차_수동.md`에 기록합니다.

### Step M-1: Playwright 리포트 열람

- **점검 목적**: AI가 실행한 Playwright 결과를 HTML 리포트에서 상세히 확인한다. 각 시나리오의 스크린샷, 실행 시간, 네트워크 요청을 시각적으로 검토하여 "PASS이지만 느리거나 불안정한 테스트"를 식별한다.
- **AI 결과 참조**: Step A-2, A-3, A-4
- **수동 확인**: `npx playwright show-report ../reports/playwright`
- **판정 기준**: 리포트 내용과 AI 결과 일치

### Step M-2: 실패 시나리오 수동 재현

- **점검 목적**: Playwright 실패 시나리오를 사람이 직접 브라우저에서 수행하여, 자동화 도구의 한계(타이밍 이슈, 동적 로딩 대기)로 인한 실패인지, 실제 기능 버그인지 구분한다.
- **AI 결과 참조**: Step A-2 FAIL 목록
- **수동 확인**: 실패한 시나리오를 브라우저에서 수동 수행
- **판정 기준**: 수동 재현 가능 여부

### Step M-3: 4개 역할별 시나리오 수동 수행

- **점검 목적**: 자동화 테스트는 미리 정의된 경로만 확인하지만, 사람은 예상치 못한 경로(뒤로가기, 새로고침, 탭 전환, 직접 URL 입력 등)를 시도할 수 있다. 각 역할로 로그인 후 핵심 플로우를 자유롭게 탐색하면서 자동화가 놓친 UX 문제를 발견한다.
- **AI 결과 참조**: Step A-2
- **수동 확인**: SRE/AI/Consultant/Agent 역할로 로그인 → 핵심 플로우 수행
- **판정 기준**: 기능 버그 또는 UX 결함 발견 건수 기록

### Step M-4: 보안 감사 결과 검토

- **점검 목적**: AI 보안 스캔은 자동화 도구 기반이므로 False Positive가 발생할 수 있다. 또한 비즈니스 로직 레벨의 취약점(권한 상승, 데이터 격리 위반)은 자동 스캔으로 탐지하기 어렵다. 사람이 결과를 검토하고, 의심 항목은 curl로 직접 재현하여 실제 위험 여부를 판단한다.
- **AI 결과 참조**: Step A-7
- **수동 확인**: 핵심 항목(인증 없는 API, XSS 가능 입력, PII 노출)을 curl/브라우저로 직접 확인
- **판정 기준**: Critical 취약점 0건

### Step M-5: 부하 테스트 결과 검토

- **점검 목적**: Locust 결과에서 P95, 실패율 등 SLO 수치를 확인하고, 특정 시나리오에서만 느린 경우(예: 수집 트리거만 P95 5000ms)를 식별한다. AI는 전체 수치를 보고하지만, 사람은 그래프의 트렌드(시간이 지나며 점점 느려지는 패턴)를 파악하여 메모리 누수 등의 장기 성능 문제를 탐지한다.
- **AI 결과 참조**: Step A-8
- **수동 확인**: Locust HTML 리포트에서 시나리오별 P50/P95/P99 확인
- **판정 기준**: P95 < 2000ms, 실패율 < 1%

### Step M-6: AI-L4/L5 분석 결과 검토

- **점검 목적**: AI의 성능 패턴 분석과 문서 일관성 분석을 사람이 검토하여, 실제로 수정이 필요한 항목과 무시해도 되는 항목을 분류한다. 예를 들어 "N+1 쿼리 의심"이라도 실제로는 캐시가 적용되어 문제가 없을 수 있다.
- **AI 결과 참조**: Step A-9, A-10
- **수동 확인**: 위험 패턴과 문서 불일치 목록 검토 → 수정 우선순위 결정
- **판정 기준**: 합의된 수정 목록 도출
