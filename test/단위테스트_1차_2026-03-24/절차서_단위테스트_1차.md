## 4. AI 실행 절차 (Claude Code)

> AI가 먼저 실행하고, 결과를 `결과서_단위테스트_1차_AI.md`에 기록합니다.

### Step A-1: 환경 검증

- **점검 목적**: 테스트 실행에 필요한 도구(Go, Node.js, npm)가 올바른 버전으로 설치되어 있는지 확인한다. 버전이 맞지 않으면 빌드 실패나 테스트 결과 불일치가 발생할 수 있으며, AI와 수동 검증의 환경 동일성을 보장하기 위해 첫 번째로 수행한다.

```bash
go version && node --version && npm --version
```

- **기대 결과**: Go 1.25+, Node 22+, npm 버전 출력
- **판정 기준**: 모든 버전이 최소 요구사항 충족 시 PASS

### Step A-2: Go 빌드 검증

- **점검 목적**: 전체 Go 소스 코드가 컴파일 에러 없이 빌드되는지 확인한다. 타입 오류, import 누락, 문법 에러 등 기본적인 코드 정합성을 보장한다. 빌드가 실패하면 테스트 자체를 실행할 수 없으므로 모든 테스트의 선행 조건이다.

```bash
cd agent && go build ./...
```

- **기대 결과**: 출력 없음 (성공)
- **판정 기준**: exit code 0이면 PASS

### Step A-3: Go 단위 테스트 실행

- **점검 목적**: 각 패키지의 함수/메서드가 설계 의도대로 동작하는지 검증한다. Collector(OS, Web, WAS, DB, Cache, MQ, LLM, OTel, VectorDB)의 수집 로직, 상태 머신 전이, RESP/Kafka 프로토콜 파싱, Heartbeat 전송, Storage 백엔드, API 계약 등 핵심 비즈니스 로직의 정확성을 확인한다. FAIL이 발생하면 해당 기능에 회귀 버그가 있다는 의미이다.

```bash
cd agent && go test -v -race -count=1 ./... 2>&1 | tee ../test/단위테스트_1차_2026-03-24/logs/go-test-output.txt
```

- **기대 결과**: 모든 패키지 `ok` 또는 `[no test files]`
- **판정 기준**: FAIL 0건이면 PASS

### Step A-4: Go 커버리지 리포트

- **점검 목적**: 테스트가 실제 코드의 몇 %를 실행하는지 측정한다. 커버리지가 낮은 패키지는 테스트가 부족하여 잠재적 버그가 숨어있을 가능성이 높다. 패키지별 수치를 기록하여 다음 차수에서 테스트를 보강할 우선순위를 결정하는 근거로 사용한다.

```bash
cd agent && go test -coverprofile=coverage.out ./... && go tool cover -func=coverage.out | tail -1
```

- **기대 결과**: 전체 커버리지 수치 출력
- **판정 기준**: 수치 기록 (임계값 없이 현황 파악)

### Step A-5: Frontend 빌드 검증

- **점검 목적**: Next.js 프레임워크의 빌드 과정에서 TypeScript 타입 에러, import 누락, 잘못된 JSX 등이 없는지 확인한다. 44개 페이지와 47개 컴포넌트가 모두 정상 컴파일되는지 검증하며, 빌드 실패 시 프로덕션 배포가 불가능하므로 Frontend 테스트의 선행 조건이다.

```bash
cd frontend && npx next build
```

- **기대 결과**: `✓ Compiled successfully`
- **판정 기준**: 빌드 성공 시 PASS

### Step A-6: Frontend Vitest 실행

- **점검 목적**: Frontend 핵심 유틸리티와 상태 관리 로직이 올바르게 동작하는지 검증한다. Button 컴포넌트(variant/size/ref/aria), i18n 다국어 번역(ko/en/ja), 숫자/날짜/바이트 포맷팅 함수, UI 스토어(사이드바/테마/로케일/시간범위/자동갱신)가 대상이다. 이 로직들은 전체 UI에서 공통으로 사용되므로 여기서 FAIL이 나면 다수 페이지에 영향을 미친다.

```bash
cd frontend && npx vitest run --reporter=verbose 2>&1 | tee ../test/단위테스트_1차_2026-03-24/logs/vitest-output.txt
```

- **기대 결과**: 모든 테스트 통과
- **판정 기준**: FAIL 0건이면 PASS

**대상 파일 (5개)**:

| # | 테스트 파일 | 대상 |
|---|-----------|------|
| 1 | `components/ui/__tests__/button.test.tsx` | Button 컴포넌트 |
| 2 | `hooks/__tests__/use-i18n.test.ts` | i18n 훅 |
| 3 | `lib/__tests__/i18n.test.ts` | i18n 라이브러리 |
| 4 | `lib/__tests__/utils.test.ts` | 유틸리티 함수 |
| 5 | `stores/__tests__/ui-store.test.ts` | UI 상태 스토어 |

### Step A-7: Frontend 커버리지 리포트

- **점검 목적**: Frontend 테스트가 실제 코드의 몇 %를 커버하는지 측정한다. vitest.config.ts에 60% 임계값이 설정되어 있으며, 이를 충족하지 못하면 테스트 안전망이 부족하다는 신호이다. 48개 페이지 중 대부분이 테스트 미비 상태이므로, 현황을 수치로 파악하여 개선 계획의 근거로 활용한다.

```bash
cd frontend && npx vitest run --coverage 2>&1 | tee ../test/단위테스트_1차_2026-03-24/logs/coverage-summary.txt
```

- **기대 결과**: 커버리지 수치 출력
- **판정 기준**: 60% 이상이면 PASS (vitest.config.ts 설정 기준)

### Step A-8: AI-L1 코드 품질 분석

- **점검 목적**: 단위 테스트로는 발견하기 어려운 코드 품질 이슈를 정적 분석으로 탐지한다. `any` 타입은 TypeScript 타입 안전성을 무력화하여 런타임 에러를 유발하고, `_ = err` 패턴은 에러를 무시하여 장애 원인 추적을 어렵게 만든다. 하드코딩 시크릿은 보안 사고로 직결된다. 이 분석은 "테스트는 통과하지만 코드가 건강한가?"를 점검한다.

Claude Code가 코드베이스를 분석하여 다음을 스캔:

| # | 점검 항목 | 대상 | 방법 | 왜 점검하는가 |
|---|----------|------|------|-------------|
| 1 | `any` 타입 사용 | `frontend/src/**/*.{ts,tsx}` | grep 스캔 | 타입 안전성 무력화 → 런타임 에러 위험 |
| 2 | 에러 무시 패턴 | `agent/**/*.go` | `_ = err` 패턴 스캔 | 에러 은폐 → 장애 진단 불가 |
| 3 | 하드코딩 시크릿 | 전체 | password, secret, api_key 리터럴 스캔 | 보안 사고 위험 (git 이력에 노출) |
| 4 | 미사용 import | `frontend/src/**/*.{ts,tsx}` | lint 기반 | 번들 크기 증가 + 코드 가독성 저하 |

- **판정 기준**: 각 항목 건수 기록 (정보 제공, 블로커 아님)

### Step A-9: AI-L2 커버리지 갭 분석

- **점검 목적**: "테스트가 아예 없는 영역"을 식별한다. 커버리지 수치가 낮은 것보다 테스트가 0건인 패키지/컴포넌트가 더 위험하다. Go 16개 패키지와 Frontend 43개 페이지에 테스트가 없으며, 이 중 config/core/database 같은 핵심 패키지의 테스트 부재는 향후 리팩토링이나 기능 추가 시 회귀 버그를 감지할 수 없다는 뜻이다. 우선순위를 정해 다음 차수에서 어디부터 보강할지 결정한다.

| # | 점검 항목 | 방법 |
|---|----------|------|
| 1 | Go `[no test files]` 패키지 | Step A-3 출력에서 추출 |
| 2 | Frontend 테스트 미비 컴포넌트 | 48 페이지 vs 5 테스트 파일 비교 |
| 3 | 우선순위 분류 | Critical/High/Medium/Low |

- **판정 기준**: 갭 목록 기록 (정보 제공)

### Step A-10: 결과서 작성

- **점검 목적**: 모든 테스트 결과를 구조화된 문서로 기록하여, 수동 검증자가 AI 결과를 재현하고 대조할 수 있도록 한다. 로그 파일 경로, 수치, PASS/FAIL 판정을 한 곳에 모아 교차검증의 기준 자료로 사용한다.

- `결과서_단위테스트_1차_AI.md` 작성
- 모든 수치/로그 참조 포함

## 5. 수동 검증 절차 (사용자)

> AI 결과를 참조하면서 수동으로 재검증합니다.
> 결과를 `결과서_단위테스트_1차_수동.md`에 기록합니다.

### Step M-1: Go 테스트 직접 실행

- **점검 목적**: AI가 보고한 테스트 결과가 환경 차이 없이 재현되는지 확인한다. AI는 자동화된 환경에서 실행하므로, 사용자의 로컬 환경(OS 설정, 환경변수, 포트 충돌 등)에서 동일한 결과가 나오는지 검증해야 한다. 양쪽 결과가 다르면 환경 의존적 버그가 존재한다는 의미이다.
- **AI 결과 참조**: Step A-3
- **수동 확인**: `cd agent && go test ./... -v` 직접 실행
- **판정 기준**: AI 결과와 PASS/FAIL 건수 일치 여부

### Step M-2: FAIL 항목 재확인

- **점검 목적**: FAIL이 발생했다면 실제 코드 버그인지, 환경 이슈(네트워크 타임아웃, 포트 충돌 등)인지 구분한다. 해당 테스트만 단독으로 재실행하여 안정적으로 재현되는지 확인하고, 간헐적 실패(flaky test)인지 판별한다. 이 단계는 "AI가 PASS라고 했지만 실제로는 불안정한 테스트"를 잡기 위한 안전망이다.
- **AI 결과 참조**: Step A-3 FAIL 목록
- **수동 확인**: FAIL 발생 시 해당 테스트만 단독 실행하여 재현 확인
- **판정 기준**: 재현 가능 여부 + 원인 분석

### Step M-3: Frontend Vitest 직접 실행

- **점검 목적**: Frontend 테스트는 jsdom 환경에서 실행되므로, Node.js 버전이나 npm 패키지 상태에 따라 결과가 달라질 수 있다. AI와 동일한 테스트를 사용자 환경에서 재실행하여 패키지 호환성 문제가 없는지 확인한다.
- **AI 결과 참조**: Step A-6
- **수동 확인**: `cd frontend && npx vitest run` 직접 실행
- **판정 기준**: AI 결과와 PASS/FAIL 건수 일치 여부

### Step M-4: 커버리지 리포트 열람

- **점검 목적**: AI가 보고한 커버리지 수치를 사람의 눈으로 검증한다. 숫자만으로는 "어떤 코드가 테스트되지 않았는지"를 알 수 없으므로, HTML 리포트에서 빨간색(미커버) 라인을 직접 확인하여 위험 영역을 파악한다. 특히 핵심 비즈니스 로직(인증, 수집, 전송)이 커버되지 않았다면 즉시 보완이 필요하다.
- **AI 결과 참조**: Step A-4, A-7
- **수동 확인**: 브라우저에서 커버리지 HTML 리포트 열기
- **판정 기준**: 수치가 AI 보고와 일치하는지 확인

### Step M-5: AI 코드 품질 결과 검토

- **점검 목적**: AI의 정적 분석은 패턴 매칭 기반이므로 False Positive(오탐)가 발생할 수 있다. 예를 들어 ECharts 라이브러리의 `any` 타입은 외부 API 제약이지 코드 결함이 아니다. 사람이 직접 코드 맥락을 확인하여 "진짜 문제"와 "의도된 패턴"을 구분해야 한다.
- **AI 결과 참조**: Step A-8
- **수동 확인**: AI가 지적한 `any` 타입, 에러 무시 위치를 직접 파일에서 확인
- **판정 기준**: 지적 내용이 실제 코드와 일치하는지 (False Positive 여부)

### Step M-6: 커버리지 갭 우선순위 합의

- **점검 목적**: AI가 식별한 테스트 미비 영역의 우선순위를 사람이 최종 결정한다. AI는 코드 구조만 보지만, 사람은 비즈니스 중요도를 안다. 예를 들어 `internal/config`는 설정 파싱이라 안정적이지만, `internal/database`는 스키마 마이그레이션 시 회귀 위험이 높다. 이런 맥락 판단은 AI가 할 수 없으므로 사람이 우선순위를 조정한다.
- **AI 결과 참조**: Step A-9
- **수동 확인**: 갭 목록을 검토하고, 다음 차수에서 보완할 우선순위 결정
- **판정 기준**: 합의된 우선순위 목록 도출
