## 1. 테스트 목표

Backend(Collection Server) + Database(PostgreSQL) + Storage(MinIO) + OTel Stack(Collector/Prometheus/Tempo/Loki) + Frontend가 Docker 환경에서 올바르게 연동되는지 검증한다. 개별 컴포넌트는 단위테스트에서 통과했으므로, 이 단계에서는 **컴포넌트 간 통신과 데이터 흐름**의 정확성을 확인한다.

## 4. AI 실행 절차 (Claude Code)

> AI가 먼저 실행하고, 결과를 `결과서_통합테스트_1차_AI.md`에 기록합니다.

### Step A-1: Go 계약 테스트 실행

- **점검 목적**: Docker 스택을 기동하기 전에, 컴포넌트 간 인터페이스(API 계약)가 올바른지 먼저 검증한다. Validation Gateway가 올바른 payload를 수용하고 잘못된 payload를 거부하는지, EventBus가 이벤트를 정확히 전파하는지, JWT 인증이 역할별(admin/sre/ai/viewer)로 동작하는지 확인한다. 이 테스트가 실패하면 Docker 스택을 기동해도 파이프라인이 동작하지 않으므로, 시간 낭비를 방지하는 게이트 역할을 한다.

```bash
cd agent && go test -v -count=1 ./test/ 2>&1 | tee ../test/통합테스트_1차_2026-03-24/logs/contract-test-output.txt
```

- **기대 결과**: 27+ 테스트 전체 PASS
- **판정 기준**: FAIL 0건이면 PASS

### Step A-2: Docker E2E 스택 기동

- **점검 목적**: Collection Server, PostgreSQL, MinIO, OTel Collector, Prometheus, Tempo, Loki, Frontend 등 전체 인프라를 실제로 기동하여 컨테이너 간 네트워크 연결, 환경변수 주입, 볼륨 마운트가 정상인지 확인한다. 컨테이너가 healthy 상태에 도달하지 못하면 이미지 빌드 오류, 포트 충돌, 의존 서비스 미기동 등의 인프라 문제가 있다는 뜻이다.

```bash
cd /c/workspace/aiservice-monitoring && docker compose -f docker-compose.e2e.yaml up -d --build 2>&1 | tee test/통합테스트_1차_2026-03-24/logs/docker-compose-up.txt
```

- **기대 결과**: 모든 서비스 started/healthy
- **판정 기준**: 전체 컨테이너 기동 성공 시 PASS

### Step A-3: 서비스 헬스체크

- **점검 목적**: 각 서비스가 기동만 된 것이 아니라, 실제 요청을 처리할 준비가 되었는지 확인한다. Collection Server `/health`, Prometheus `/-/ready`, Tempo `/ready`, Loki `/ready`, OTel Collector `/`, MinIO `/minio/health/live`, Frontend `localhost:3000` 각각의 헬스 엔드포인트를 호출하여 응답을 검증한다. 헬스체크 실패 시 해당 서비스의 로그를 확인하여 근본 원인을 파악해야 한다.

```bash
bash scripts/e2e/healthcheck.sh 2>&1 | tee test/통합테스트_1차_2026-03-24/logs/healthcheck-output.txt
```

- **기대 결과**: 전체 서비스 PASS
- **판정 기준**: FAIL 0건이면 PASS

### Step A-4: 파이프라인 검증 (21개 체크포인트)

- **점검 목적**: AITOP의 핵심 데이터 플로우가 end-to-end로 동작하는지 검증한다. Agent가 등록(heartbeat) → 데이터 수집(collect) → 서버 수신 → 저장(MinIO/PostgreSQL) → 조회(Fleet API) → 메트릭 노출(Prometheus)까지의 전체 경로를 실제로 실행한다. 이 파이프라인이 끊어지면 모니터링 솔루션의 핵심 기능이 동작하지 않는다.

```bash
bash scripts/phase17-3/02-pipeline-verify.sh 2>&1 | tee test/통합테스트_1차_2026-03-24/logs/pipeline-verify-output.txt
```

검증 항목:

| # | 체크포인트 | 점검 내용 |
|---|-----------|----------|
| P1 | Health Check | 서버 기동 상태 |
| P2 | JWT 인증 (4건) | login, me, refresh, 역할별 토큰 발급 |
| P3 | Heartbeat (2건) | Agent 등록 + Fleet 반영 확인 |
| P4 | Collect Result (3건) | OS/LLM/Diagnostic 수집 데이터 제출 |
| P5 | SSE EventBus | 실시간 이벤트 스트리밍 |
| P6 | MinIO S3 (4건) | 버킷 생성 + 객체 저장 확인 |
| P7 | Fleet API (2건) | 에이전트 목록/상세 조회 |
| P8 | 수집 트리거 (2건) | 수동 수집 명령 실행 |
| P9 | Prometheus (2건) | 메트릭 수집 + 쿼리 응답 |

- **기대 결과**: 21/21 PASS
- **판정 기준**: FAIL 0건이면 PASS, 1건 이상 FAIL이면 CONDITIONAL PASS + 원인 분석

### Step A-5: UI API 검증

- **점검 목적**: Frontend가 호출하는 API 엔드포인트가 Backend에서 실제로 응답하는지 확인한다. Phase 17 테스트에서 12건의 FAIL(미구현 API)이 발견되었으며, 이 엔드포인트들이 현재 구현되었는지 또는 여전히 미구현 상태인지 추적한다. 미구현 API는 Frontend에서 에러 화면이 표시되므로, 사용자 경험에 직접 영향을 미친다.

```bash
bash scripts/phase17-3/03-ui-api-verify.sh 2>&1 | tee test/통합테스트_1차_2026-03-24/logs/ui-api-verify-output.txt
```

Phase 17 기준 Known FAIL (미구현 API):

| 엔드포인트 | 상태 | 영향 |
|-----------|------|------|
| `/infra/hosts` | 미구현 | 인프라 호스트 목록 페이지 |
| `/fleet/agents/{id}` | 미구현 | 에이전트 상세 조회 |
| `/ai/services`, `/ai/gpu` | 미구현 | AI 서비스 대시보드 |
| `/ai/services/{id}/llm,rag,guardrail` | 미구현 | AI 서비스 상세 |
| `/diagnostics/*` | 미구현 | 진단 보고서 |

- **기대 결과**: PASS/FAIL 건수 기록 + Known FAIL 변동 추적
- **판정 기준**: 핵심 API(auth, heartbeat, collect, fleet) PASS이면 CONDITIONAL PASS

### Step A-6: AI-L3 API 호환성 분석

- **점검 목적**: Frontend 소스 코드에서 호출하는 API 경로와 Backend 라우터에 등록된 핸들러를 정적으로 대조한다. 실제 HTTP 요청 없이도 "Frontend가 `/api/v1/agents`를 호출하는데 Backend에는 `/api/v1/fleet/agents`로 등록되어 있다"와 같은 경로 불일치를 발견할 수 있다. 이런 불일치는 런타임에서 404/405 에러로 나타나며, Docker 스택 없이도 탐지할 수 있다.

- Frontend `fetch`/`axios` 호출 경로 추출
- Backend 라우터 핸들러 경로 추출
- 양쪽 경로 대조 → 불일치 목록 작성

- **판정 기준**: 불일치 건수 기록 (정보 제공)

### Step A-7: Docker 스택 정리

- **점검 목적**: 테스트 완료 후 Docker 리소스(컨테이너, 볼륨, 네트워크)를 정리하여 디스크/메모리를 확보한다. `-v` 플래그로 볼륨도 삭제하여 다음 테스트 실행 시 클린 상태를 보장한다.

```bash
docker compose -f docker-compose.e2e.yaml down -v 2>&1
```

- **기대 결과**: 전체 컨테이너/볼륨 제거
- **판정 기준**: 에러 없이 완료 시 PASS

## 5. 수동 검증 절차 (사용자)

> AI 결과를 참조하면서 수동으로 재검증합니다.
> 결과를 `결과서_통합테스트_1차_수동.md`에 기록합니다.

### Step M-1: Docker 스택 직접 기동

- **점검 목적**: AI 환경과 동일한 Docker 스택이 사용자 환경에서도 정상 기동되는지 확인한다. Docker Desktop 버전, 디스크 여유 공간, 포트 충돌 등 환경 차이로 인한 문제를 탐지한다.
- **AI 결과 참조**: Step A-2, A-3
- **수동 확인**: `docker compose -f docker-compose.e2e.yaml up -d --build` 실행 후 `docker compose ps`로 상태 확인
- **판정 기준**: 모든 서비스 healthy

### Step M-2: 핵심 API 수동 호출

- **점검 목적**: curl로 핵심 엔드포인트를 직접 호출하여 응답 형식과 데이터를 눈으로 확인한다. AI는 HTTP 상태 코드만 확인하지만, 사람은 응답 JSON의 내용이 의미적으로 올바른지(예: 에이전트 목록에 실제 에이전트가 있는지) 판단할 수 있다.
- **AI 결과 참조**: Step A-4
- **수동 확인**:
  ```bash
  # Health
  curl http://localhost:8080/health
  # Login
  curl -X POST http://localhost:8080/api/v1/auth/login -d '{"email":"admin@aitop.io","password":"admin"}'
  # Agents
  curl -H "Authorization: Bearer {token}" http://localhost:8080/api/v1/fleet/agents
  ```
- **판정 기준**: 응답 JSON 형식 정상 + 데이터 의미 확인

### Step M-3: Frontend UI 직접 접속

- **점검 목적**: 브라우저에서 실제 UI를 열어 페이지가 렌더링되는지, 데이터가 표시되는지, 에러 화면이 나타나지 않는지 확인한다. API가 정상이어도 Frontend 렌더링 버그(하이드레이션 오류, CSS 깨짐, 빈 데이터 표시)가 있을 수 있으며, 이는 브라우저에서만 확인 가능하다.
- **AI 결과 참조**: Step A-5
- **수동 확인**: `http://localhost:3000` 접속 → 데모 계정 로그인 → 핵심 페이지 10개 탐색
  - `/` (홈), `/infra`, `/services`, `/traces`, `/ai`, `/agents`, `/alerts`, `/slo`, `/settings`, `/topology`
- **판정 기준**: 페이지 렌더링 정상 + 에러 콘솔 없음

### Step M-4: Docker 로그 확인

- **점검 목적**: 서비스가 healthy 상태이더라도, 로그에 경고(WARN)나 반복적인 에러가 있을 수 있다. 예를 들어 "connection refused" 재시도가 반복되면 서비스는 결국 연결되지만 시작 시간이 길어진다. 이런 숨겨진 문제를 로그에서 탐지한다.
- **AI 결과 참조**: 전체
- **수동 확인**: `docker compose -f docker-compose.e2e.yaml logs --tail=50` 에서 ERROR/WARN 확인
- **판정 기준**: Critical 에러 0건

### Step M-5: AI-L3 API 불일치 검토

- **점검 목적**: AI가 보고한 Frontend↔Backend API 경로 불일치를 직접 확인하고, 의도된 차이(예: 프록시 경로 변환)와 실제 버그를 구분한다.
- **AI 결과 참조**: Step A-6
- **수동 확인**: 불일치 목록에서 핵심 항목을 실제 코드에서 확인
- **판정 기준**: 실제 버그인 불일치 건수 기록
